# CLAUDE.md ŌĆö Constellation

## Project Purpose

Integrative bioinformatics platform: a shared core of physically grounded primitives (compositions, distributions, optimizers, graphs) with modality modules (mass spectrometry, sequencing, structure, codon optimization, NMR) layered on top. The goal is to connect experimental modalities through common abstractions ŌĆö mathematical structures exposed once in `core/` and reused everywhere ŌĆö rather than siloed per-modality code.

Constellation is absorbing four existing lab packages as a **clean rewrite** (no vendored source): **Cartographer** (MS proteomics, first port target), **NanoporeAnalysis** (POD5 ŌåÆ protein), **CoLLAGE** (codon optimization), **Contour** (PDB ŌåÆ MD prep). Chronologer (RT prediction) has already been absorbed into Cartographer.

Current state: **`core.chem`, `core.sequence`, `core.io`, `core.structure`, `core.stats`, `core.optim`, a minimal `core.graph.Network`, `massspec.peptide` (first domain-module port target), and `massspec.{library, quant, acquisitions}` (theoretical / empirical / provenance trio) shipped; everything else scaffold.** Functional code lives in `constellation.core.chem` (full periodic table, Composition arithmetic, binned + isotopologue-resolved isotope distributions, UNIMOD vocabulary with subsetting), `constellation.core.sequence` (canonical + IUPAC alphabets, generic ops, nucleic translation/ORF/rev-complement, protein cleavage / peptide composition), `constellation.core.io` (`RawReader` ABC + suffix/modality registry, `Bundle`/`OpcBundle`/`DirBundle` for multi-file containers, canonical Arrow schemas `Trace1D` / `SpectraMatrix2D` / `PeakTable` with namespaced metadata extras, torch-tensor bridges), `constellation.core.graph` (Arrow-backed `Network[NodeT, EdgeT]` with neighbor lookup / induced subgraphs / connected components), `constellation.core.structure` (`AtomTable` Arrow schema for per-atom identity, separate `(F, N, 3)` torch coord tensor on `Ensemble`, `Topology` with bond/angle/dihedral graphs and `infer_bonds` from covalent radii, composable `pa.compute.Expression` selection helpers, tensor-native geometry ŌĆö translate / rotate / centroid / mass_centroid / radius_of_gyration / principal_axes; Kabsch / RMSD / superposition deferred until `core.optim` ships), `constellation.core.stats` (two-tier ABC: `Parametric(nn.Module)` with `.fit(data, *, optimizer, ...)` driver, `Distribution` requires `.log_prob` + `.cdf`, `PeakShape` requires `.integrate` + `.bounds`; 9 distributions, 2 peak shapes, 3 calibration models, 5 losses, CODATA units; consumes `core.optim.Optimizer` / `PopulationOptimizer` Protocols and dispatches on optimizer shape), `constellation.core.optim` (`Optimizer` + `PopulationOptimizer` Protocols, `LBFGSOptimizer` with bounds-clamping, `DifferentialEvolution` with vectorized population eval via `vmap` + `functional_call`, 6 mutation strategies, binomial/exponential crossover, Sobol/LHS/uniform init, F/CR dithering, optional terminal LBFGS polish reusing `LBFGSOptimizer`, `OPTIMIZER_REGISTRY`), `constellation.thirdparty.registry` (tool discovery), and `constellation.cli.__main__` (the `constellation doctor` command). All other subpackages are scaffold-only. Core primitives and domain modules are staged for the 90-day roadmap (see `docs/roadmap.md` when written).

## Architecture invariants (load-bearing ŌĆö design principles from the PI)

1. **Physically grounded statistical models** with explicit priors ŌĆö preferred over heuristics.
2. **HUPO-PSI standards compliance is the default** for proteomics-domain serialization, identifiers, and metadata keys. Where a HUPO-PSI standard exists for a concept Constellation handles, the PSI form is the canonical *external* representation; relational Arrow tables remain the canonical *in-memory* shape (PSI-bridge, not PSI-first). In-scope: **ProForma 2.0** (modseq strings ŌĆö `core.sequence.proforma`), **mzPAF v1.0** (peak annotations ŌĆö `massspec.annotation.mzpaf`), **mzSpecLib v1.0** (spectral library exchange ŌĆö `massspec.library.readers.mzspeclib`, Phase 3), **USI** (spectrum identifiers ŌĆö `massspec.annotation.usi`), **PSI-MS CV** (metadata-key vocabulary ŌĆö `core.ontology.PSI_MS`, Phase 2). PSI-MS CV accessions (`MS:NNNNNNN`) are the preferred metadata-key form anywhere a CV term exists; the `x.<domain>.<key>` namespace remains the fallback for Constellation-specific keys with no PSI equivalent. Deferred until corresponding readers/writers drive them: **mzML**, **mzIdentML**, **mzTab**, **SDRF-Proteomics**, **PEFF**, **mzPeak** (when ratified), **mzQC**, **proBAM / proBed** (lands with the genomeŌåÆtranscriptomeŌåÆproteome bridge work). Out of scope: **PSI-MI XML**, **TraML**, **FeatureTab**. New PSI ratifications enter the in-scope set unless a written rationale rejects them.
3. **PyArrow tables in memory, Parquet on disk.** No pandas inside the package; pandas exits at the port boundary and stays at the edge.
4. **PyTorch for numerical work** wherever possible ŌĆö multithreaded performance + optimizers + ML.
5. **Generic, nested abstractions** ŌĆö trees, networks, composition tensors, distributions, optimizers expose statistical similarities across modalities that are otherwise treated as orthogonal.
6. **Unified treatment of optimizers and distributions.** Peak shapes and probability densities share a single `Parametric` ABC; `DifferentialEvolution` and `torch.optim.*` share one registry.
7. **Close-to-instrument data access.** Provide readers that sit directly above the vendor/instrument format; document the assumptions each processing layer adds.
8. **CLI-first** (single `constellation <subcommand>` binary with thin legacy shims for Cartographer muscle memory); GUI later.
9. **Linux primary, WSL second, macOS best-effort.**
10. **Dual audience** ŌĆö importable library for Jupyter-notebook lab research + polished external-user tool.

### Code-level invariants

- **PascalCase for all new classes.** Legacy `snake_case` class names from Cartographer (`resnet_block`, `resnet_unit`) do not come across.
- **No pandas inside the package.** Arrow in, Arrow out.
- **Degenerate-alphabet compatibility rule.** `core.sequence.alphabets` supports IUPAC degenerate codes (N, R, Y, B, Z, X, U, O, ...). Canonical alphabets expose chemical compositions; degenerate alphabets do not. Functions gate via `requires_canonical` / `degenerate_ok`; composition/mass/cleavage reject degenerate tokens, while kmerize/reverse-complement/ORF tolerate them. `translate()` resolves degenerate codons by enumerating canonical expansions and collapsing the residue set ŌĆö wobble-position synonymy stays unambiguous (`CTN ŌåÆ L`), two-residue ambiguities map to `B`/`Z`/`J`, anything else (including stop+coding mixes) collapses to `X`.
- **Pluggable codon tables.** `translate(seq, codon_table=STANDARD)` accepts any entry from `CODON_TABLES` (keyed by NCBI transl_table number). Shipped: 1 (Standard), 2 (Vertebrate Mito), 3 (Yeast Mito), 4 (Mold/Protozoan/Coelenterate Mito + Mycoplasma), 5 (Invertebrate Mito), 6 (Ciliate), 11 (Bacterial/Archaeal/Plastid). Stored in `constellation/data/codon_tables.json` as overrides on top of table 1.
- **Cleavage uses regex, not pyteomics.** `core.sequence.protein.cleave` is hand-rolled with the third-party `regex` module (overlap-aware). The 17 built-in protease patterns (Trypsin family, LysC/N, ArgC, Chymotrypsin, AspN, GluC, Pepsin, ProteinaseK, Thermolysin, No_enzyme) are lifted from ExPASy PeptideCutter and ship in `constellation/data/proteases.json`. Pyteomics still earns its place later in `massspec` for mzML I/O ŌĆö but `core.sequence` does not import it.
- **Third-party tool env-var convention:** `$CONSTELLATION_<TOOL>_HOME` (uniform across all tools ŌĆö no `_JAR` / `_DIR` / `_HOME` mixing).
- **Generic-vs-modality split for schemas.** A schema earns a slot in `core.io.schemas` only if its column layout is genuinely universal across Ōēź2 modalities (`Trace1D`, `SpectraMatrix2D`, `PeakTable` qualify). Modality-specific information rides as **schema metadata, never as columns**. Preferred key form: **PSI-MS CV accession** (`MS:1000744` for "selected ion m/z", `MS:1000041` for "charge state", etc.) when one exists for the attribute (per architectural invariant #2); fallback to namespaced `x.<domain>.<key>` (`x.hplc.wavelength_nm`, `x.ce.capillary_index`, `x.pod5.read_id`, `x.constellation.fragment_ladder_version`) for Constellation-specific keys with no PSI equivalent. So `Trace1D` is `(time_s, intensity)` everywhere; HPLC reads `x.hplc.wavelength_nm`, CE reads `x.ce.capillary_index`, POD5 reads `x.pod5.read_id`. Producers stamp the metadata; downstream math doesn't care which modality the bytes came from. Round-trip metadata through `pack_metadata` / `unpack_metadata` from `core.io.schemas` to keep the JSON encoding consistent.
- **EncyclopeDIA modseq parsing at the reader boundary.** EncyclopeDIA's `[+N.NNN]` mass-delta notation is ambiguous between N-terminal ╬▒-amine and residue side-chain placement (e.g. `K[+42.011]VERPD` could be K side-chain Acetyl OR N-terminal Acetyl on K ŌĆö these are chemically distinct molecules). `constellation.massspec.io.encyclopedia.parse_encyclopedia_modseq` resolves this by looking up each mass delta in UNIMOD and applying the modification's `specificities` constraints: terminal-only mods at residue 0 promote to `[Mod]-X` form; mods that fit both placements default to side-chain unless localization evidence overrides. Mass deltas that don't resolve unambiguously pass through as ProForma 2.0 mass-delta `ModRef` entries (no silent misassignment). The function returns a `Peptidoform` directly so callers don't double-parse for downstream fragment annotation; `format_proforma(peptidoform)` populates `PEPTIDE_TABLE.modified_sequence`. The inverse `format_encyclopedia_modseq(peptidoform)` is the writer-side helper ŌĆö lossy on the chemical-fidelity axis because EncyclopeDIA cannot represent terminal vs side-chain placement (terminal mods collapse onto residue 0). The translation lives in the encyclopedia format adapter at `massspec.io.encyclopedia`, not in `core.chem` and not in `thirdparty/` ŌĆö it's pure file-format translation, not a tool-jar binding.
- **Torch-first numerics; numpy only at I/O boundaries.** Per Principle 3, `core.signal`, `core.stats`, and `core.optim` operate on `torch.Tensor` end-to-end. Numpy is permitted only at: (a) big-endian byte decode where torch lacks the dtype (FA `.raw` ``>u2``) ŌĆö one boundary-crossing per file via `torch.from_numpy(arr.copy())`; (b) the Arrow Ōåö torch bridge (`pa.Array.to_numpy(zero_copy_only=True)` + `torch.from_numpy(...)` is zero-copy in both legs). Little-endian dtypes decode straight through `torch.frombuffer` without numpy.
- **`core.stats` parameterization conventions.** Positive parameters live in log-space (`log_sigma`, `log_tau`, `log_N_max`) and bounded parameters in logit-space (`logit_eta`, mixture weights) so any unconstrained gradient-based optimizer can fit them cleanly. Default dtype is float64 ŌĆö peak-shape and density numerics are precision-sensitive. `Parametric.parameters_dict()` decodes back to natural units for inspection. DE bounds operate on the registered name (e.g. `log_sigma`, not `sigma`); use `core.optim.bounds_in_natural_units(model, {"sigma": (1e-3, 1e3)}, log_params=["sigma"])` to convert from natural units ŌĆö there is no name-prefix auto-detection.
- **Optimizer Protocols (`core.optim`).** `Optimizer.step(closure) -> Tensor` ŌĆö closure returns un-backed loss; the optimizer (LBFGSOptimizer, torch.optim.LBFGS) handles `zero_grad` + `backward` internally. `PopulationOptimizer` adds `is_converged` and uses a vectorized closure `closure(model_view) -> (pop_size,) loss`, where `model_view` is built by DE from `torch.func.vmap(torch.func.functional_call(...))`. Implication: every `Parametric` subclass's `forward` (and `log_prob` / `cdf` for `Distribution`) must be `vmap`-compatible ŌĆö basic broadcasting torch ops are fine; boolean indexing with dynamic shape (`x[mask] = ...`) breaks vmap and must be rewritten with `torch.where`. `Parametric.fit(data, *, optimizer, ...)` dispatches on `isinstance(opt, PopulationOptimizer)`; an optional `polish_on_converge=True` flag triggers `optimizer.polish(scalar_closure)` for optimizers that expose it (DE).
- **CDF gap (pinned).** `StudentT.cdf` and `Beta.cdf` raise `NotImplementedError` in v1 ŌĆö torch lacks `betainc` as of 2.11, a scipy bridge would break autograd, and a custom autograd Function is heavier than the v1 use case justifies. Lift when torch ships `betainc` or when a concrete gradient-aware-CDF use case appears (Cartographer error-model port goes through `log_prob` end-to-end and is unaffected). All other CDFs are pure torch.
- **Cartographer scoring inheritance.** `core.stats.losses` mirrors Cartographer's `kld` / `spectral_angle` formulae but as **free functions** (Cartographer's `KLD` / `Spectral_Entropy_Loss` `nn.Module` wrappers used `snake_case` class names against the PascalCase invariant). `spectral_angle` clamps the dot product to `[-1, 1]` instead of `[-1+╬Ą, 1+╬Ą]` ŌĆö the latter pulls identical-vector similarity below 1 by ~3e-4 in float32. `spectral_entropy_loss` completes the truncated Cartographer implementation (return statement was commented out upstream).

## Import order ŌĆö project-wide DAG

Load-bearing invariant. Violating it creates circular imports or module-init ordering bugs. Enforced by convention, not linting.

### Inside `core/`

```
chem ŌöĆŌ¢║ sequence ŌöĆŌ¢║ {graph, nn, optim} ŌöĆŌ¢║ stats ŌöĆŌ¢║ signal ŌöĆŌ¢║ structure
                                                                Ō¢▓
                                  io ŌŚäŌöĆŌöĆ (leaf, consumed by all)ŌöĆŌöś
```

Tertiary structure / dynamics are *consequences* of chemistry + sequence + (graph) topology + thermodynamics + optimization, not building blocks for them ŌĆö so `core.structure` sits at the bottom of the math chain. Bond graphs route through `core.graph`; non-trivial structural manipulation (alignment, fitting, constrained geometry) wants the objective-function + optimizer machinery in `core.stats` + `core.optim`.

- `chem` ŌĆö never imports from `sequence`, `structure`, or `io`.
- `sequence` ŌĆö imports `chem`; never imports `structure`.
- `graph` ŌĆö generic over node/edge types, independent of `chem` and `sequence`.
- `nn` ŌĆö downstream math primitives; depends on `torch` only.
- `optim` ŌĆö leaf-tier; depends on `torch` only. Ships the `Optimizer` and `PopulationOptimizer` Protocols, `LBFGSOptimizer` (production wrapper around `torch.optim.LBFGS` with bounds clamping), `DifferentialEvolution` (vectorized via `vmap` + `functional_call`), and `OPTIMIZER_REGISTRY`. Operates generically on any `nn.Module`.
- `stats` ŌĆö imports `core.optim` for Protocols and dispatches `Parametric.fit()` on optimizer shape (scalar vs population). May import from `chem`, `sequence`. Do NOT import `structure`.
- `signal` ŌĆö heuristic 1D-trace operations (baseline, smoothing, peak picking, polynomial calibration). Imports `stats` for parametric peak-fitting helpers; otherwise leaf-style. Distinct from `stats` because the operations have imprecisely-specified priors and don't earn membership in the `Parametric` namespace.
- `structure` ŌĆö most-downstream core module. May import any of `chem`, `sequence`, `graph`, `stats`, `nn`, `optim` as features land. Currently imports just `chem`, `sequence`, `graph`, `io`; `stats` / `optim` enter when Kabsch / RMSD / refinement land in `core.structure.geometry`.
- `io` ŌĆö leaf. Format/codec concerns only. Never imports other core. The `AtomTable` schema self-registers with the `core.io` schema registry from `core.structure.atoms`.

### Between layers

```
core  ŌöĆŌ¢║  {massspec, sequencing, structure, codon, nmr,
           chromatography, electrophoresis}  ŌöĆŌ¢║  cli
           (domain modules ŌĆö never cross-import)
```

Domain modules import from `core` and from `thirdparty` adapters; they **never cross-import to another domain module**. If a workflow legitimately spans modalities (e.g. the transcriptomic ŌåÆ proteomic pipeline), it lives as a thin top-level script under `constellation/` that imports from the relevant domains ŌĆö no dedicated `bridges/` or `pipelines/` folder until we have Ōēź2 such workflows to compare.

Adding a new module: confirm its place in this DAG before touching imports. Adding a new reader (Bruker `.d`, Sciex `.wiff`, DCD trajectory, ...) follows the reader-subclasses-core.io.readers rule without special handling.

## Environment

Conda env `constellation`; Python 3.12 only (pinned `<3.13` by **koinapy 0.0.10 + PyTorch 2.5.x** ŌĆö same ceiling Cartographer uses). `torch`, `koinapy`, and `pythonnet` are all pip-installed via the `pip:` block in `environment.yml` ŌĆö PyPI's default `torch` wheel is CUDA-capable, which works on CPU-only hosts and lets GPU nodes (OSC Cardinal, etc.) use CUDA without changing env.yml.

**Additive CUDA overlay:** `environment-cuda.yml` exists as a stub; apply on top of the base env on GPU hosts when GPU-specific deps (dorado builds, CUDA mmseqs) get wired in.

## Conventions

- **Functions:** `snake_case`.
- **Classes:** `PascalCase` (always ŌĆö no legacy carryover).
- **Modification identifiers:** UNIMOD format (`UNIMOD:N`) is canonical; mass notation (`+N.NNN`) is a secondary representation. The full UNIMOD XML ships built-in (`UNIMOD: ModVocab`); EncyclopeDIA-style `[+mass_delta]` translation goes through `constellation.massspec.io.encyclopedia.parse_encyclopedia_modseq(modseq) -> Peptidoform` (which calls `UNIMOD.find_by_mass(mass, tolerance_da)` and applies specificity constraints to disambiguate N-terminal vs side-chain placement). The translation lives in the encyclopedia format adapter at `massspec.io.encyclopedia`, not in `core.chem` and not in `thirdparty/` ŌĆö the function is pure format translation, not a tool-jar binding. All EncyclopeDIA readers MUST route incoming modseq strings through this helper; the resulting `Peptidoform` serializes via `format_proforma(p)` for `PEPTIDE_TABLE.modified_sequence`.
- **ProForma 2.0 is the canonical modseq representation.** HUPO-PSI ProForma 2.0 (Final draft 15, Feb 2022; LeDuc et al. *J. Proteome Res.* 21:1189ŌĆō1195, 2022; spec PDF + EBNF at github.com/HUPO-PSI/ProForma) parser/formatter live in [`core.sequence.proforma`](constellation/core/sequence/proforma.py): `parse_proforma(s) -> Peptidoform | MultiPeptidoform`, `format_proforma(p)`. `PEPTIDE_TABLE.modified_sequence` strings are ProForma 2.0 strings. The migration off the old `parse_modified_sequence` was driven by a chemistry-fidelity bug ŌĆö `K[Acetyl]VERPD` (lysine ╬Ą-amine acetylation) and `[Acetyl]-KVERPD` (N-terminal ╬▒-amine acetylation) are isobaric but chemically distinct molecules with different RT/fragmentation/localization-scoring behavior; they must be distinguishable in-memory. Downstream consumers (`peptide_composition`, `peptide_mass`, `precursor_mz`, `peptide_envelope`, `fragment_ladder`) all take `Peptidoform` directly ŌĆö bare-string callers wrap with `Peptidoform(sequence="PEPTIDE")`. **pyteomics is BANNED from `core.sequence.proforma`**: the parser is independently implemented from the upstream EBNF, and pyteomics is used only in cross-validation tests + the future `massspec` mzML reader.
- **`Peptidoform.charge` is a serialization artifact, not a chemical property.** The ProForma 2.0 spec encodes precursor charge as a `/N` suffix on the modseq string (and adducts as `/N[+Na+,-H+]`), so a peptidoform parsed from a library record or search-engine output carries the published charge state alongside the residue sequence and modifications. The molecule itself isn't `+2` in solution ŌĆö that charge state is acquired in the ESI source. `precursor_mz(peptidoform, *, charge=None, ...)` falls back to `peptidoform.charge` when the kwarg is omitted but lets explicit `charge=N` override (e.g. for charge-state grids over a charge-naive peptidoform). If we ever need to separate the format-shaped charge from the molecule-shaped charge, that's a `PrecursorRecord`-style wrapper on top, not a refactor of `Peptidoform`.
- **Cleavage output is NŌåÆC ordered, deduplicated.** `cleave()` returns peptides sorted by `(start, end)`, with sequence-level deduplication keeping first occurrence (pyteomics returns a `set` ŌĆö random order). `return_spans=True` yields `Peptide(sequence, start, end, n_missed)` records with parent-protein coordinates intact, no dedup. `cleave_sites()` always anchors on `[0, ..., len(seq)]`.
- **Peptide mass uses heavy-isotope correction.** `peptide_mass(peptidoform)` adds `mod.delta_composition` to the peptide composition for all mods; for mods carrying `mass_override` (TMT10, SILAC ┬╣┬│CŌéå, ...) the difference `mod.mass_override - mod.delta_composition.mass` is added on top so the result matches the canonical UNIMOD value. Light-skeleton compositions remain in the returned `Composition`; the heavy-isotope correction lives in the mass calculation, not in the composition itself. Global isotope labels (`<13C>`, `<15N>`, ...) follow the same pattern: `peptide_mass` substitutes the natural-abundance monoisotopic mass for the labeled isotope's exact mass via `Element.isotope_mass(N)`; the composition's element counts are unchanged.
- **Vocabulary vs enablement.** `core.chem.modifications` defines the *universe* (built-in `UNIMOD`); each downstream tool declares its *enabled subset* via `UNIMOD.subset([...])` and ships that with its model checkpoint / pipeline config. The chemistry layer never knows what any specific tool accepts. `register_custom(...)` is the typed escape hatch for genuinely novel non-UNIMOD mods; the legacy `user_modifications.txt` text format from cartographer/Chronologer is **not** carried forward.
- **Composition is plain-Python, not a torch.Tensor subclass.** Hashable, equality-as-bool, only chemistry-meaningful ops exposed. Batched / GPU work uses raw `(B, N_elements)` tensors via `stack()` / `batched_mass()` free functions ŌĆö `Composition` objects are not in the hot path.
- **Isotope data carries per-isotope exact masses (NIST AME2020), not just abundances.** `isotope_distribution` / `isotope_envelope` use the binned ┬╣┬│C-spacing approximation (cartographer-equivalent, fast); `isotopologue_distribution` / `isotope_envelope_exact` resolve ┬╣ŌüĄN vs ┬╣┬│C as distinct peaks for high-res MS, ┬╣ŌüĄN-labeled samples, and NMR-adjacent work.
- **Composition tensor dtype is int32.** Counts are always whole; `.mass`/`.average_mass` cast to float64 internally for the dot-product. Averaged "compositions" (mixtures) are a downstream concept and are NOT a `Composition`.
- **Tolerance:** default fragment tolerance 20 ppm; functions accept `tolerance_unit` as `'ppm'` or `'Da'` (same as Cartographer).
- **Sentinel values:** `-1.0` = "not observed" / "not set" for float fields; `-1` in intensity tensors flags loss-masked positions.
- **Spectral libraries split theoretical from empirical.** `massspec.library` carries the protein ŌåÆ peptide ŌåÆ precursor ŌåÆ fragment DAG plus the only true M:N edge (protein Ōåö peptide) and entity-level *predicted* properties (RT / CCS / fragment intensity) ŌĆö sample-agnostic, sequence-derivable. `massspec.quant` carries per-acquisition observations (Protein/Peptide/PrecursorQuant) and *calibrated* per-edge transmission efficiencies (`TransmissionProteinPeptide`, `TransmissionPeptidePrecursor`); `efficiency` is strictly bounded `(0, 1]` on populated rows with `-1.0` as the uncalibrated sentinel. `massspec.search` holds peptide-level + protein-level detection scores (q-values, posterior error probabilities, percolator scores) and is the third sibling ŌĆö keyed by `(entity_id, acquisition_id, engine)`. Run provenance lives once in `massspec.acquisitions` (`ACQUISITION_TABLE` + `Acquisitions` container + `validate_acquisitions` cross-check); Library, Quant, and Search all reference `acquisition_id`. A higher-level composition layer that unifies Library + Search + Quant per Constellation's "detection-IS-quantification" philosophy is intentionally deferred until a concrete pipeline drives its API. Library/Quant/Search each ship Reader/Writer Protocols + registries with `ParquetDirReader/Writer` as the native lossless form. EncyclopeDIA `.dlib` / `.elib` (the same SQLite container ŌĆö DLIB and ELIB differ only in which tables are populated) is implemented in `massspec.io.encyclopedia` and self-registers with all three Reader/Writer registries; one shared core produces a `(Library, Quant?, Search?)` tuple via `read_encyclopedia(path)` and slot-specific Protocols (`load_library` / `load_quant` / `load_search`) project out the requested view.
