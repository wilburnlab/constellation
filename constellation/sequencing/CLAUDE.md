# CLAUDE.md — `constellation.sequencing`

This file extends the project-wide rules in [../../CLAUDE.md](../../CLAUDE.md). Read that first.

`sequencing/` absorbs **NanoporeAnalysis** as a clean rewrite. The lab's workflows span genomic DNA (POD5 → Dorado → HiFiASM → polish → scaffold → annotate), transcriptomic cDNA (segmented edlib demux → ORF prediction → kmer-cluster consensus → predicted proteins feeding `massspec` libraries), and direct-RNA / modifications. Rather than mirroring `massspec`'s theoretical/empirical/scoring trio, sequencing's natural shape is `acquisitions + samples + reads + alignments + (genome reference, annotation, transcript reference) + assembly + quant + annotation tools + genetic_tools` — see the conventions section below for the rationale.

## Module index

**S1 transcriptomics demux pipeline shipped:** `constellation transcriptome demultiplex --reads ... --library-design cdna_wilburn_v1 --samples ... [--min-aa-length 60] [--threads N] [--batch-size 100000] [--resume] [--progress]` runs SAM/BAM → READ_TABLE → READ_SEGMENT_TABLE / READ_DEMUX_TABLE / ORF_TABLE → Protein × Sample count matrix. Stages: `_plan_chunks` (parent-only chunk-spec computation: BAM record-boundary virtual offsets via fast no-decode pysam scan, SAM newline-aligned byte ranges) → `run_fused_demux_stage` (N parallel workers each open their own pysam handle on a slice of the source file, do ingest + `locate_segments` + `predict_orfs` + `resolve_demux` (sample_id resolved locally via the Samples container) + `build_partial_quant` end-to-end, write 5 shards: reads / read_segments / read_demux / orfs / feature_quant; multi-file inputs allowed — `--reads file1.bam file2.bam` or `--reads dir/` with a 'file' column in the samples TSV mapping each (sample, barcode) edge to its source acquisition) → `resolve` (streaming aggregator: `aggregate_partial_quants` opens `feature_quant/` as a `pa.dataset` and group_by-sums into the final small matrix; emits `feature_quant.parquet` + `proteins.fasta` + `protein_counts.tsv`). No single-file `read_demux.parquet` aggregation — `read_demux/` stays a partitioned dataset queryable via `pa.dataset.dataset(read_demux/)`. Per-worker file slicing eliminates the parent-serial ingest bottleneck — at `--threads 96` on a cluster node, the BAM is decoded 96 ways in parallel.

`HardThresholdScorer` matches NanoporeAnalysis `_fixed1` semantics with three deliberate accuracy improvements over the upstream baseline (filter edlib `-1` no-match sentinel; short-circuit on empty subjects; skip the literal-string `cDNA[1] == 'Unknown'` check that hides real annotations behind `'Unknown Fragment'`). Parity vs lab `_fixed1` baseline: 99.5% per-read status agreement, 99.78% barcode agreement, **100% count-matrix per-cell parity, 100% protein-FASTA sequence-set match**.

Other shipped: `Acquisitions` / `Samples` containers (validators-only mode), `cdna_wilburn_v1` design (24 barcodes, regenerable via `scripts/build-sequencing-primers-json.py`), `align/{locate,pairwise}` (edlib-backed `locate_substring` + `pairwise_align`), `Scorer` Protocol seam (`HardThresholdScorer` shipped is the production scorer; `ProbabilisticScorer` deferred indefinitely — see scoring-domain rule below), `transcriptome.simulator` (synthetic-read corpus generator: `ReadSpec` + `generate_stress_test_specs` + `simulate_panel` writes SAM + ground-truth parquet for stress-testing the scorer + as a regression fixture for cross-Dorado-version drift), `scripts/analyze-demux-statistics.py` (dataset-level filter + Arrow join + numpy histograms; per-status edit-distance + polyA-length distributions; **diagnostic tool**, not a calibration target — used for spotting basecaller drift across Dorado versions, characterising per-barcode systematics, regression-testing scorer changes).

19 self-registering Arrow schemas including `READ_TABLE` extended with `dorado_quality` (qs:f), `read_group` (RG:Z), `duplex_class` (dx:i, int8 — 0=simplex/1=duplex/-1=simplex-with-duplex-offspring) so the Dorado tags useful for QC + S3 duplex-artifact validation are captured at S1 ingest; `READ_DEMUX_TABLE` extended in S1 with `status` (string), `is_fragment` (bool), `artifact` (string, populated `"none"` until S3); `READ_SEGMENT_TABLE.score_delta` (nullable int32; best-vs-second-best gap on barcode rows).

**Reference layer shipped (S1 reference data classes):** the `Reference` stub split into three sibling Library-shaped containers — `GenomeReference` (`CONTIG_TABLE` + `SEQUENCE_TABLE`), `Annotation` (`FEATURE_TABLE`, FK closure into a `GenomeReference` via `validate_against`), `TranscriptReference` (`TRANSCRIPT_TABLE`; `from_annotation(genome, annotation)` materialises spliced transcripts via strand-aware exon concatenation against contig sequences). Each ships PK/FK closure on construction + ParquetDir Reader/Writer Protocols. `GeneticTools` is the cRAP-equivalent for genomics — 67 curated entries across 9 categories at `constellation/data/genetic_tools.json`, regenerable via `scripts/build-genetic-tools-json.py`; `load_default_genetic_tools()` is the bundle loader. Hand-rolled stdlib-only readers: `readers/fastx.py::read_fasta_genome` / `read_fasta_transcriptome` / `read_fasta_genetic_tools`, `readers/gff.py::read_gff3` / `write_gff3` (two-pass parent resolution, gzip-aware, multi-parent expansion, `##FASTA` boundary), `readers/bed.py::read_bed`. `FEATURE_QUANT` extended with a `feature_origin` column (vocabulary `'gene_id'` | `'transcript_id'` | `'cluster_id'` | `'genetic_tool'` | `'protein_uniref'` | `'protein_hash'`); `transcriptome.quant.to_feature_quant` projects the existing naive-ORF output into the canonical schema under `feature_origin='protein_hash'`. CLI wired: `constellation reference {import, fetch, summary, validate}`; `fetch` uses stdlib `urllib.request` against Ensembl / Ensembl Genomes / RefSeq URL conventions (no NCBI Datasets CLI dep). `Assembly.to_reference` renamed to `to_genome_reference`; companion Annotations build from downstream BUSCO / repeat / telomere passes (still stubbed).

**Mode A genome-guided gene counting shipped (S2):** `constellation transcriptome align --demux-dir <s1-out> --reference <ref-root> --samples <tsv> --output-dir <dir> [filter knobs] [--resume] [--progress]` chains directly off the S1 demux output (reads input BAM paths from the demux manifest). Three stages: (1) `align` — `map_to_genome` materialises a cached genome FASTA + `.mmi` index from the `GenomeReference`, runs minimap2 splice-aware (`-ax splice -uf --cs=long --secondary=no`) per input file, then `samtools cat | samtools sort | samtools index` into one coordinate-sorted `bam/aligned.bam`. (2) `decode_filter_overlap` — `_bam_alignment_chunks` plans virtual-offset chunks; the fused worker (`fused_decode_filter_overlap_worker`) decodes each chunk via `read_bam_alignments_chunk`, applies `apply_filter_predicates` (length / aligned-fraction / mapq / primary-only — vectorised pa.compute), then `compute_gene_overlap` (sort+searchsorted sweep-join against the broadcast gene set, best-overlap selection, antisense gating). Each worker writes 4 partitioned shards: `alignments/`, `alignment_tags/`, `gene_assignments/`, `stats/`. Worker-local alignment_id packed as `(worker_idx << 32) | local_idx` for collision-free partition concatenation. (3) `count` — `count_reads_per_gene` materialises `gene_assignments/` + `read_demux/` as pa.Tables (bounded ~3 GB / ~10 GB at 200M-read scale), hash-joins on `read_id`, group_by-sums to `FEATURE_QUANT(feature_origin='gene_id')`, and populates `tpm` per-sample as `count × 1e6 / sum_in_sample(count)` — long-read TPM convention (one read = one transcript, no length normalisation). The CLI then writes wide gene × sample TSVs alongside the canonical parquet via `quant.gene_matrix.build_gene_matrix` + `render_gene_matrix_tsv`: `gene_counts.tsv` and `gene_tpm.tsv` (controlled by `--matrix-format {count,tpm,both,none}` default `both`, with `--matrix-min-count N` to drop low-total rows; gene rows carry feature_id / gene_name / gene_biotype / contig / start / end / strand / length annotation columns + one float column per sample, with zero-fill for genes absent from `feature_quant` for stable cross-run indexing). Single output entry point (`pa.Table` inputs) serves both pipeline (CLI materialises from datasets) and tests/Jupyter (callers pass small in-memory tables).

**Streaming pipeline rule:** the in-memory `Alignments` container + `Alignments.filter` (with per-stage `metadata_extras['filter_steps']` audit) + `count_reads_per_gene` API are for tests + Jupyter only — at 30–200M alignments per real run (Pichia ~30M, mouse/Arabidopsis ~200M), the pipeline never materialises a full Alignments. The fused-worker streaming + bounded-resolve-stage memory pattern is the only production path. **CLI namespace rule:** gene-counting is a transcriptomics analysis (consumes a reference; not a reference-management op), so it lives at `transcriptome align`, not `reference align`. Mode B (de novo clustering) shares the same verb when it lands — dispatch on `--reference` presence; absent → "not yet implemented" with exit 2. **Two-layer minimap2 split:** `align/minimap2.py::minimap2_run(target, queries, *, output_path, args, threads)` is the generic, use-case-agnostic subprocess wrapper (knows nothing about splice / preset); `align/map.py::map_to_genome` is the use-case orchestrator that hard-codes the splice flags. Future `map_assembly` / `map_dna_to_genome` / PAF-output verbs all compose `minimap2_run` directly with their own arg lists — no subclassing or wrapping. mappy is rejected (user-confirmed slower); the `map_reads` two-backend stub is deleted. **Memory budget for the resolve stage:** `gene_assignments` ~3 GB + `read_demux` ~10 GB at 200M reads — fits comfortably on workstation RAM. If we cross 1B reads, `count_reads_per_gene` lifts into an Arrow-Dataset-streaming variant — single-function refactor at that point, not a new public API.

**Phase 1 alignment-derived intermediates shipped (power-user toggles on `transcriptome align`, off by default — no cost on the existing fast path):** three new schemas in `schemas/alignment.py` — `ALIGNMENT_BLOCK_TABLE` (one row per CIGAR-derived M/=/X-contiguous block; cs:long-derived `n_match`/`n_mismatch` populated when minimap2 emitted `--cs=long`, null when only CIGAR is available; contig comes via the `alignment_id` FK to `ALIGNMENT_TABLE.ref_name` to save a column × ~600M rows at PromethION scale), `SPLICE_JUNCTION_TABLE` (cross-read aggregate keyed on `(contig_id, donor_pos, acceptor_pos, strand)` with `read_count`, `motif` from genome at donor[0:2]/acceptor[-2:], optional `annotated` when an Annotation is supplied), `READ_FINGERPRINT_TABLE` (per-read canonical splicing-topology hash via stdlib `blake2b` over `(contig_id, strand, [(donor//q, acceptor//q)...])`; primary alignments only; this is the Phase 2 cluster key). Producers: `align/cigar.py::parse_cigar_blocks` + `parse_cs_long_blocks` (state-machine parsers for both grammars; block-break rule is N + D≥`intron_min_bp` but never long I — long insertions add bases to the query within the same aligned region); `quant/genome_count.py::extract_alignment_blocks` (worker-layer orchestrator; transparently captures the cs tag when needed and strips it from the user-visible tag table); `quant/coverage.py::build_pileup` (sweep-line over per-block intervals → RLE depth via depth-transition emission only, so half-open abutments coalesce; per-sample stratification via `read_to_sample` mapping with `sample_id=-1` sentinel for the unstratified mode); `quant/junctions.py::aggregate_junctions` (cross-read junction reduction with strand-naive +-strand motif lookup); `transcriptome/fingerprints.py::compute_read_fingerprints`. CLI flags on `transcriptome align`: `--emit-blocks`, `--emit-pileup`, `--emit-junctions`, `--emit-fingerprints` (the latter three imply `--emit-blocks`), `--intron-min-bp` (default 25), `--intron-quantum-bp` (default 10). Wired through the existing fused worker (gated `emit_blocks` kwarg → fifth `alignment_blocks` partitioned shard) + new resolve-stage aggregators that materialise `alignment_blocks/` + `alignments/` once and emit `coverage.parquet` / `junctions.parquet` / `read_fingerprints.parquet`.

**Why aggregate introns rather than exons in `SPLICE_JUNCTION_TABLE`:** splice sites are *steep* (canonical GT-AG dinucleotide-anchored) but not strictly *discrete* — cryptic-splice excursions are real and we deliberately preserve them as a basecaller / spliceosome-fidelity diagnostic (the `--intron-quantum-bp` knob absorbs *small* excursions into the same fingerprint bucket while letting larger ones surface as cluster discordance). The per-read exon view already lives in `ALIGNMENT_BLOCK_TABLE`; cross-read exon aggregation (a `CANDIDATE_EXON_TABLE`) is deferred to gene-structure inference (roadmap Layer 3) where it's the natural primitive for unannotated-genome exon discovery.

**Phase 2 genome-guided fingerprint clustering shipped:** `constellation transcriptome cluster --mode genome-guided --align-dir <s2-out> --demux-dir <s1-out> --samples <tsv> --output-dir <dir> [--build-consensus] [--max-5p-drift 25] [--max-3p-drift 75] [--intron-quantum-bp 10] [--min-cluster-size 1] [--per-sample-clusters] [--write-fasta] [--write-summary] [--threads N] [--resume] [--progress]` ships an isoform-resolution clusterer that consumes Phase 1's `alignment_blocks/` (NOT the optional `read_fingerprints.parquet` — fingerprints are derived fresh at clustering time so `--intron-quantum-bp` is a *clustering* parameter, not an alignment parameter; sweeping the quantum never requires re-running align). Algorithm: (1) re-derive fingerprints by calling `compute_read_fingerprints(alignment_blocks=..., intron_quantum_bp=N)` against the partitioned `alignment_blocks/` dataset, (2) group by `fingerprint_hash` per (contig, strand[, sample_id]), (3) strand-aware 5'/3' span-coherence drift filter — on `+`, 5' = `span_start`; on `-`, 5' = `span_end`; singletons skip the filter (no median to compare against), (4) Layer-0 dereplicate trimmed transcript-window sequences (blake2b-8 hash of `READ_TABLE.sequence[transcript_start:transcript_end]`) per cluster and pick the most-abundant unique sequence's read as representative (tiebreak by mean `dorado_quality`), (5) optional `--build-consensus` per-position weighted-PWM majority vote on the genome coordinate frame via the shared `align/consensus.py` kernel. Two new schemas in `schemas/transcriptome.py`: `TRANSCRIPT_CLUSTER_TABLE` extended in place with `mode` ∈ `{'genome-guided', 'de-novo'}` discriminator + `contig_id`/`strand`/`span_start/end`/`fingerprint_hash`/`n_unique_sequences`/`sample_id` (Phase 3 de novo will populate `consensus_sequence` + `predicted_protein` on the same row shape), and `CLUSTER_MEMBERSHIP_TABLE` (long-form (cluster_id, read_id) edges with `role` ∈ `{'representative', 'member', 'duplicate', 'drift_filtered'}` + multi-parameter fit-to-cluster scores: signed `drift_5p_bp` / `drift_3p_bp` from cluster median + cs:long-derived `match_rate` / `indel_rate` / `n_aligned_bp` per primary alignment). New modules: `transcriptome/cluster_genome.py::cluster_by_fingerprint` (worker), `align/consensus.py::build_consensus` (torch-first weighted-PWM via `scatter_add_` on a `(L, 5)` float64 tensor — A/C/G/T/gap; walks cs:long in lockstep with `alignment_blocks` to project per-base substitution / deletion votes onto reference positions; insertions and splice operators advance the read pointer without contributing votes; gap-winning positions drop from the consensus). Phase 1 retrofit: `extract_alignment_cs(alignments, tags) -> ALIGNMENT_CS_TABLE` + new `--emit-cs-tags` flag on `transcriptome align` writes a sixth `alignment_cs/` partitioned shard preserving per-alignment cs:long strings (the worker already captures cs transparently for block extraction; the retrofit adds a user-visible sidecar — required by `--build-consensus` to skip the prohibitively-slow re-run-NW-per-member alternative). Drift-filtered reads are *kept* in `cluster_membership` with `role='drift_filtered'` by default (Phase 4 cross-validation needs to ask "where did Phase 3 land these?"); `--drop-drift-filtered` opts out. Cluster-pair merge for subset-junction-noise (truncated reads with the same internal splicing as a longer cluster's full intron chain) is deferred to v2 — Phase 4 empirical signal will tell us how often truncations split clusters in practice.

**Granularity contract for `transcriptome cluster --mode genome-guided`:** clustering is **per-isoform**, not per-gene. The `fingerprint_hash` keys on `(contig_id, strand, [(donor//q, acceptor//q)...])` so different splicing topologies on the same gene fall into different clusters; reads agreeing on quantised splicing collapse into one cluster regardless of substitution noise (within-cluster heterogeneity surfaces via `n_unique_sequences > 1` and per-membership `match_rate` < 1.0). A gene with N annotated isoforms produces ≥ N clusters in v1 — the inequality picks up novel isoforms (correctly distinct), truncated reads that miss an internal junction (incorrectly split; deferred subset-junction merge is the v2 fix), and basecaller-induced junction-call errors (rare; surfaces as singletons). Alt-TSS / alt-polyA isoforms whose intron chain is identical fall into the *same* fingerprint cluster initially and the drift filter separates the long-tail divergent reads — Phase 2 doesn't subcluster within a fingerprint group; that's Phase 3 / 4 territory.

**Still stubs:** `Assembly` / `Project` container method bodies, `Dorado` runner, `transcriptome/cluster.py` / `consensus.py` re-exports for the Phase 3 de novo pipeline, `--mode de-novo` and `--mode validate` branches of `transcriptome cluster`, assembly / polish / scaffold / annotation runners. CLI wired: `transcriptome demultiplex` (S1) + `transcriptome align` (S2 Mode A, with Phase 1 power-user `--emit-{blocks,pileup,junctions,fingerprints,cs-tags}` toggles) + `transcriptome cluster --mode genome-guided` (Phase 2) shipped.

## Conventions (sequencing layer)

- **Sequencing containers do NOT mirror the massspec Library/Quant/Search trio.** Sequencing's natural shape is `acquisitions + samples + reads + alignments + (genome reference, annotation, transcript reference) + assembly + quant + annotation tools + genetic_tools`. The MS theoretical/empirical split doesn't transfer because for de novo work the "library" *is* the experimental output. The reference layer is **split into three sibling Library-shaped containers** because real-world references are heterogeneous: `GenomeReference` (`CONTIG_TABLE` + `SEQUENCE_TABLE`), `Annotation` (`FEATURE_TABLE`, FK into a `GenomeReference`), `TranscriptReference` (`TRANSCRIPT_TABLE`; populated either from a paired GenomeReference + Annotation via `TranscriptReference.from_annotation` or from a standalone transcript FASTA when no genome exists). Each ships PK/FK closure + ParquetDir native form + Reader/Writer Protocols. `GeneticTools` is the cRAP-equivalent for genomics — a curated bundle of common cloning / engineering parts (AbR / FPs / tags / promoters / common enzymes / vector backbones) at `constellation/data/genetic_tools.json`. `Assembly` is its own container with provenance Reference doesn't need (read-coverage, polishing rounds, scaffolding edges) and `Assembly.to_genome_reference()` lifts a finalized assembly into the same `GenomeReference` shape so external-ref and de-novo-ref workflows share downstream code; companion Annotations build separately from BUSCO / RepeatMasker / telomere passes. `Quant` keys per-feature counts/coverage/TPM on `(feature_id, sample_id, engine, feature_origin)` — multi-tag, see below. `Alignments` is its peer container (per-acquisition, FK to `GenomeReference`). No `Search` analog — alignments are observations, not detection scores.

- **`feature_quant` is multi-tag, not strict-precedence.** `FEATURE_QUANT` carries a `feature_origin` column with vocabulary `'gene_id'` | `'transcript_id'` | `'cluster_id'` | `'genetic_tool'` | `'protein_uniref'` | `'protein_hash'` (frozen set in `sequencing.schemas.quant.FEATURE_ORIGIN_VOCAB`). `feature_id` is unique within a `feature_origin` partition; sums within a partition are valid counts; **never sum across partitions** (different statistical models of "what a feature measures"). The transcriptome S1 naive-ORF output projects into FEATURE_QUANT under `feature_origin='protein_hash'` via `transcriptome.quant.to_feature_quant`; future genome-aligned gene counts use `'gene_id'`; transcriptome-mode EM lands `'transcript_id'`; mmseqs2 hits land `'genetic_tool'` or `'protein_uniref'`.

- **External-format readers are hand-rolled, stdlib only.** `readers/fastx.py::read_fasta_genome` / `read_fasta_transcriptome` / `read_fasta_genetic_tools`, `readers/gff.py::read_gff3` / `write_gff3`, and `readers/bed.py::read_bed` are pure-Python streaming parsers (gzip-aware via stdlib). No `pyranges`, no `gffutils`, no pandas. Reasoning: pyranges1 pulls pandas + a gtfreader sub-dep + a Rust backend; pysam.FastaFile only earns its keep on huge genomes that need `.fai`-indexed random contig access (we do full one-pass reads). Same precedent as `readers/sam_bam.py` — pysam stays scoped to where its CIGAR / virtual-offsets / sorting machinery is genuinely irreplaceable. Coordinates are 0-based half-open internally; GFF3 (1-based inclusive) and BED (already 0-based half-open) convert at the reader boundary.
- **`Samples` is first-class in sequencing, not derived.** A sample = `(acquisition_ids, barcode_id?)` resolved via the M:N `SAMPLE_ACQUISITION_EDGE` table. Single-sample multi-flowcell (genomic ultra-HMW; barcode null) and multi-sample single-flowcell (multiplexed transcriptomic; M barcodes) and mixed designs all use the same schema — no bifurcation by experiment type.
- **Generic `ragged_trace_1d` schema covers POD5 + future MS scan blobs.** Variable-length 1D-array-per-row with rich row-level metadata earns a slot in `core.io.schemas` (≥2 modalities). POD5 reads are 1k-5M+ samples each (~10¹³ rows if flattened to `Trace1D`); MS scans are 10⁴-10⁵ peaks each. The factory takes a value dtype + optional explicit-x column + per-row extras: POD5's `RAW_SIGNAL_TABLE` instantiates with `value_dtype=int16, value_name='signal'`, future massspec scans with `value_dtype=float64, value_name='intensity', x_name='mz'`. `ragged_to_tensors` projects the ragged columns to per-row torch tensors (each row may have a different length — caller pads / batches).
- **Demux output is segment-row-shaped, not read-row-shaped.** `READ_SEGMENT_TABLE` carries one row per *located segment* (adapter / polyA / barcode / transcript / UMI) — chimeric / concatemer reads (very common in nanopore) emerge as multiple segment-runs per read without any chimera-aware special casing. Demux logic operates on `LibraryDesign(layout: tuple[Segment, ...])` composed of `AdapterSlot` / `BarcodeSlot` / `PolyASlot` / `UMISlot` / `TranscriptSlot`; new ONT/SMARTer-derived chemistries land as new designs without code changes. `Adapter` and `Barcode` are independent registry-grade objects (kits share adapters across products). `READ_DEMUX_TABLE` is the derived view: one row per resolved transcript window with `sample_id` assigned and `is_chimera` flagged.
- **Sequencing demux uses segmented edlib over Dorado's full-primer Smith-Waterman.** Three-step algorithm (lifted from NanoporeAnalysis): (1) anchor on polyA via error-tolerant 'AAAA' edit-distance scan, (2) locate 3' PCR adapter on polyA-side; score the gap region against barcode panel, (3) locate 5' adapter and isolate transcript window. Beats Dorado's built-in demux because high homopolymer error in nanopore A-runs (lab measurement: mean A=26 against expected A=20) distorts SW scoring of the full primer enough to misassign barcodes; segmenting on polyA first and scoring only the discriminating region sidesteps this.
- **`Scorer` Protocol — pluggable per-segment scoring; `HardThresholdScorer` is production.** `transcriptome.scoring.Scorer` is the seam between hard-threshold and probabilistic scoring of segments (polyA / 5' adapter / 3' adapter / barcode panel). The orchestrator (`transcriptome.demux.locate_segments`) calls `Scorer` methods uniformly — swapping modes is a one-line change at the CLI / API surface, not a fork in the demux algorithm. `BarcodeVerdict.delta_edit_distance` is the gap between the best and second-best matches across the FULL panel (not just within-budget candidates), so `READ_SEGMENT_TABLE.score_delta` is directly useful as a diagnostic — a 0-vs-1 best-second gap is much weaker evidence than a 0-vs-6 gap even though both look like "Complete" matches under the hard threshold. **`ProbabilisticScorer` is deferred indefinitely** per the scoring-domain rule below; the seam stays in place at zero cost so a future basecaller emitting trustworthy posteriors can drop in without re-architecting demux.

- **Scoring-domain rule: don't dissect basecaller artifacts in the demux scorer.** `HardThresholdScorer` matches NanoporeAnalysis `_fixed1` semantics with three deliberate accuracy improvements over the upstream baseline and is the production scorer. The originally-planned `ProbabilisticScorer` (S2) was meant to fit calibrated `Parametric` distributions over polyA length / adapter edit distance / barcode panel scores, with priors learned from the empirical distributions in `analyze-demux-statistics.py`. **We deferred this work after the polyA-by-barcode diagnostic showed that per-barcode polyA-length systematics are dominant and motor-coupled** — see `data/sequencing/cdna_wilburn_v1.json::calibration_notes` (BC06 example: TAT-repeat structure interacts with the motor protein during the polyA dwell, biasing the basecaller's dwell-time → A-count conversion). Calibrating a probabilistic demux scorer to the basecaller's per-barcode artifacts would be curve-fitting to a black box rather than recovering signal. The principled path lives one layer earlier: (a) **signal-level demux** working on the raw POD5 squiggle directly (change-point detection on dwell-time series, not sequence similarity on basecalled output — same domain as `signal.models.physical` for direct-RNA modifications), or (b) **better basecaller training** with constellation supplying the trustworthy-ground-truth fixtures (synthetic-read corpus + lab-measured systematics) a training run needs. The `Scorer` Protocol seam stays — drop-in for whichever path matures first — but no `Parametric` calibration work happens against current Dorado output.

- **The simulator + stats infrastructure is a diagnostic tool, not a calibration substrate.** `transcriptome.simulator` and `scripts/analyze-demux-statistics.py` were originally designed to feed S2's prior calibration. After the scoring-domain pivot above, they keep their value — for spotting basecaller drift across Dorado versions (re-run the stats script on a new Dorado release; if BC06's polyA mean shifts, the basecaller's homopolymer model changed), regression-testing `Scorer` implementations against the synthetic ground truth, and characterising per-(library design × basecaller) systematics. They are NOT a substrate for fitting empirical priors that pretend to recover ground truth from current Dorado output.
- **HardThresholdScorer corrects three NanoporeAnalysis bugs deliberately.** (1) Filter edlib's `editDistance=-1` no-match sentinel before picking a winning barcode (NanoporeAnalysis's filter checks `is not None`, which leaks `-1`s and assigns no-match reads to whichever barcode is first in dict-iteration order). (2) Short-circuit on empty `ref` strings in `locate_substring` — edlib on an empty subject returns `editDistance=len(query)` with `locations=[(None, -1)]`, which NanoporeAnalysis treats as a "match" and ends up assigning bogus 16-edit-distance "matches" against empty putative_umi regions. (3) Pick the non-Unknown orientation cleanly when exactly one orientation classifies as Unknown — NanoporeAnalysis's `cDNA[1] == 'Unknown'` literal-string check fails when reverse is `'Unknown Fragment'` (not just `'Unknown'`) and silently throws away forward's real annotation. All three are documented in code where the divergence happens; constellation's behavior is strictly more accurate than the upstream baseline.
- **PolyA-barcode boundary ambiguity is a scoring concept, not a panel-data property.** When a barcode's RC begins with 'A' (BC05/06/11/12 in `cdna_wilburn_v1`), polyA detection greedily absorbs the leading A — the barcode matches the panel at ed=1 instead of ed=0. The leading A is information-theoretically null under hard-mode scoring (no upper bound on polyA length means no signal in length variation). The Pichia-flowcell diagnostic (calibration_notes in `cdna_wilburn_v1.json`) checked whether stratifying polyA length by leading-A-RC-class recovers a clean bimodal structure that a probabilistic scorer could exploit — answer was **no** (means within ~0.1 nt of each other; per-barcode systematics like BC06's TAT-repeat motor effect are larger and unrelated to leading-A class). The boundary-absorption hypothesis is documented but does NOT appear to be the dominant source of polyA-length variation. Per the scoring-domain rule above, never strip from the panel + never trim at scorer setup — the per-barcode polyA-length empirical distribution stays intact as a Dorado-behaviour diagnostic, and downstream consumers that care about boundary ambiguity address it at the basecaller level (signal-domain demux), not the demux scorer.
- **S1 stage parallelism: per-worker file chunks, fused ingest+demux, partitioned-parquet shards.** `sequencing.parallel.run_batched` is the generic batch executor: each worker writes its own `<output>/<key>/part-NNNNN.parquet` shard directly (no Arrow tables pickled back to the parent), `_SUCCESS` markers gate stage-level resume, missing shards inside an unfinished stage trigger shard-level partial recovery. `transcriptome.stages.run_demux_pipeline` orchestrates `_plan_chunks` (parent-only; cheap chunk-spec computation) → `run_fused_demux_stage` (N parallel workers each do ingest + demux + ORF) → `resolve` (sequential aggregation). For BAM, the planner does a fast no-decode pysam scan (`_bam_record_chunks` in `readers/sam_bam.py`) recording record-boundary virtual offsets every `chunk_size` primary records; for SAM, byte-range chunks split at newlines via the worker-side handshake protocol (each worker discards a partial first line if not at file start, then reads complete lines until past `byte_end`). Each worker opens its own pysam handle on the source file, so ingest is fully N-way parallel — no parent-serial bottleneck. Replaces NanoporeAnalysis transcriptome_nf (Nextflow per-batch fan-out → physical SAM shards on disk) with `--threads N` + `--batch-size` + `--resume` CLI flags. Worker function (`transcriptome.stages._fused_chunk_worker`) lives at module top level so `ProcessPoolExecutor` can pickle it; designs are loaded by NAME inside workers (cheap JSON read; cached) rather than pickled across the IPC boundary. Each worker emits 4 shards: `reads`, `read_segments`, `read_demux`, `orfs` — the `reads` dataset is partitioned (one parquet per worker), queryable via `pa.dataset.dataset(reads/)`.
- **Progress reporting is a Protocol, not stdout calls.** `sequencing.progress.ProgressCallback` accepts `ProgressEvent` (kind: `stage_start` / `stage_progress` / `stage_done`, plus `stage`, `completed`, `total`, `message`, free-form `payload`). CLI binds `StreamProgress` (one line per event to stderr) under `--progress`; future GUI binds the same Protocol to a Qt signal or websocket. The core verbs in `parallel.run_batched` and `transcriptome.stages` take `progress_cb: ProgressCallback | None` and emit at well-known points without caring how the host renders them.
- **`align/` modules named by verb, not backend.** `pairwise_align(query, ref, *, backend='edlib'|'parasail')`, `map_reads(reads, reference, *, backend='mappy'|'minimap2')`, `locate_substring(query, ref, *, max_distance)`. Backend choice is a kwarg, not visible at the import level — keeps demux / polish / scaffold callers clean and lets us swap implementations without touching call sites.
- **`DoradoModel` parses lab shorthand.** `DoradoModel.parse("sup@v5.0+5mC,5hmC")` → `DoradoModel(family='sup', version='v5.0', chemistry='dna_r10.4.1_e8.2', mods=('5mC', '5hmC'))`. Default chemistry is DNA R10.4.1; explicit chemistry prefix (`"rna004_130bps_sup@v5.0.0+m6A_DRACH"`) handles direct-RNA. `model_name()` renders to Dorado's canonical on-disk model directory name; `modified_bases_arg()` formats for the `--modified-bases` CLI flag.
- **`DoradoRunner` returns `RunHandle`, never blocks.** Single-flow-cell sup-model basecalling takes 3-4 days on an A100 — long enough that blocking the Python session is wrong. `RunHandle` exposes `wait()` / `poll()` / `tail_progress()` / `cancel()`, and `RunHandle.attach(pid_file)` rebinds to a previously-detached run from a different Python session (so `constellation basecall --follow` can monitor a job started days earlier).
- **Pure-functional first; `Project` is the orchestrator on top.** Verbs like `bam_dir_to_assembly(bam_paths)` work standalone with no project context. `Project.open(root)` reads a `manifest.toml`, knows the standard layout (`pod5/ → bam/ → assembly/ → annotation/ → transcriptome/`), and is the *only* layer that mutates filesystem state — same pattern as `Library` / `save_library`. Labs that produce BAMs externally and just want the analysis layer skip the Project entirely.
- **Cross-modality workflows live as top-level scripts (one workflow today).** `constellation/transcriptome_to_proteome.py` is the only such workflow today, so it lives at top-level per the "≥2 workflows before a folder" rule. When a second cross-modality pipeline lands (genome → spectral library, structure → spectral prediction), they co-locate into a dedicated subpackage; name TBD (`bridges/` and `pipelines/` are ruled out — candidates: `asterism/`, `confluence/`, `weft/`, `synthesis/`, `composer/`, `lattice/` — defer until two workflows exist to compare).
