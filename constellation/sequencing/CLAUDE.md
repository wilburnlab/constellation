# CLAUDE.md — `constellation.sequencing`

This file extends the project-wide rules in [../../CLAUDE.md](../../CLAUDE.md). Read that first.

`sequencing/` absorbs **NanoporeAnalysis** as a clean rewrite. The lab's workflows span genomic DNA (POD5 → Dorado → HiFiASM → polish → scaffold → annotate), transcriptomic cDNA (segmented edlib demux → ORF prediction → kmer-cluster consensus → predicted proteins feeding `massspec` libraries), and direct-RNA / modifications. Rather than mirroring `massspec`'s theoretical/empirical/scoring trio, sequencing's natural shape is `acquisitions + samples + reads + alignments + (genome reference, annotation, transcript reference) + assembly + quant + annotation tools + genetic_tools` — see the conventions section below for the rationale.

## Module index

**S1 transcriptomics demux pipeline shipped:** `constellation transcriptome demultiplex --reads ... --library-design cdna_wilburn_v1 --samples ... [--min-aa-length 60] [--threads N] [--batch-size 100000] [--resume] [--progress] [--emit-fastq]` runs SAM/BAM → READ_TABLE → READ_SEGMENT_TABLE / READ_DEMUX_TABLE / ORF_TABLE → Protein × Sample count matrix. The `--samples` TSV format is **header-required**, columns `sample_name + barcode_id` (plus optional `file` for multi-file mode, `description`); `sample_id` is auto-assigned per unique `sample_name` so rows sharing a `sample_name` aggregate into one sample with multiple `(acquisition, barcode)` edges — that's how multi-barcode multiplexing and multi-flowcell aggregation are expressed. The resolved `Samples` container is persisted under `<output-dir>/samples/` (ParquetDir bundle: `samples.parquet` + `edges.parquet` + `manifest.json`) so downstream stages (`transcriptome align`, `transcriptome cluster`) load it back via `load_samples(...)` and **do not require `--samples` to be re-supplied**. Stages: `_plan_chunks` (parent-only chunk-spec computation: BAM record-boundary virtual offsets via fast no-decode pysam scan, SAM newline-aligned byte ranges) → `run_fused_demux_stage` (N parallel workers each open their own pysam handle on a slice of the source file, do ingest + `locate_segments` + `predict_orfs` + `resolve_demux` (sample_id resolved locally via the Samples container) + `build_partial_quant` end-to-end, write 5 shards: reads / read_segments / read_demux / orfs / feature_quant; multi-file inputs allowed — `--reads file1.bam file2.bam` or `--reads dir/` with a 'file' column in the samples TSV mapping each (sample, barcode) edge to its source acquisition) → `resolve` (streaming aggregator: `aggregate_partial_quants` opens `feature_quant/` as a `pa.dataset` and group_by-sums into the final small matrix; emits `feature_quant.parquet` + `proteins.fasta` + `protein_counts.tsv`). No single-file `read_demux.parquet` aggregation — `read_demux/` stays a partitioned dataset queryable via `pa.dataset.dataset(read_demux/)`. Per-worker file slicing eliminates the parent-serial ingest bottleneck — at `--threads 96` on a cluster node, the BAM is decoded 96 ways in parallel. Optional `--emit-fastq` adds a fourth stage (`transcriptome/fastq.py::emit_per_sample_fastq`) that streams `read_demux/` ⨝ `reads/` (transcript-trimmed window, strict Complete-only filter — same predicate as `_iter_demux_read_batches(only_complete=True)`) and routes each batch to per-sample `gzip.open` handles, emitting `fastq/<sample_name>.fq.gz` (aggregated across acquisitions/barcodes that share `sample_id`); atomic `.tmp → .fq.gz` rename + own `fastq/_SUCCESS` marker make it resume-safe and a clean bolt-on for users re-running the same command with `--emit-fastq --resume` on an existing output-dir.

`HardThresholdScorer` matches NanoporeAnalysis `_fixed1` semantics with three deliberate accuracy improvements over the upstream baseline (filter edlib `-1` no-match sentinel; short-circuit on empty subjects; skip the literal-string `cDNA[1] == 'Unknown'` check that hides real annotations behind `'Unknown Fragment'`). Parity vs lab `_fixed1` baseline: 99.5% per-read status agreement, 99.78% barcode agreement, **100% count-matrix per-cell parity, 100% protein-FASTA sequence-set match**.

Other shipped: `Acquisitions` / `Samples` containers (validators-only mode), `cdna_wilburn_v1` design (24 barcodes, regenerable via `scripts/build-sequencing-primers-json.py`), `align/{locate,pairwise}` (edlib-backed `locate_substring` + `pairwise_align`), `Scorer` Protocol seam (`HardThresholdScorer` shipped is the production scorer; `ProbabilisticScorer` deferred indefinitely — see scoring-domain rule below), `transcriptome.simulator` (synthetic-read corpus generator: `ReadSpec` + `generate_stress_test_specs` + `simulate_panel` writes SAM + ground-truth parquet for stress-testing the scorer + as a regression fixture for cross-Dorado-version drift), `scripts/analyze-demux-statistics.py` (dataset-level filter + Arrow join + numpy histograms; per-status edit-distance + polyA-length distributions; **diagnostic tool**, not a calibration target — used for spotting basecaller drift across Dorado versions, characterising per-barcode systematics, regression-testing scorer changes).

19 self-registering Arrow schemas including `READ_TABLE` extended with `dorado_quality` (qs:f), `read_group` (RG:Z), `duplex_class` (dx:i, int8 — 0=simplex/1=duplex/-1=simplex-with-duplex-offspring) so the Dorado tags useful for QC + S3 duplex-artifact validation are captured at S1 ingest; `READ_DEMUX_TABLE` extended in S1 with `status` (string), `is_fragment` (bool), `artifact` (string, populated `"none"` until S3); `READ_SEGMENT_TABLE.score_delta` (nullable int32; best-vs-second-best gap on barcode rows).

**Reference layer shipped (S1 reference data classes):** the `Reference` stub split into three sibling Library-shaped containers — `GenomeReference` (`CONTIG_TABLE` + `SEQUENCE_TABLE`), `Annotation` (`FEATURE_TABLE`, FK closure into a `GenomeReference` via `validate_against`), `TranscriptReference` (`TRANSCRIPT_TABLE`; `from_annotation(genome, annotation)` materialises spliced transcripts via strand-aware exon concatenation against contig sequences). Each ships PK/FK closure on construction + ParquetDir Reader/Writer Protocols. `GeneticTools` is the cRAP-equivalent for genomics — 67 curated entries across 9 categories at `constellation/data/genetic_tools.json`, regenerable via `scripts/build-genetic-tools-json.py`; `load_default_genetic_tools()` is the bundle loader. Hand-rolled stdlib-only readers: `readers/fastx.py::read_fasta_genome` / `read_fasta_transcriptome` / `read_fasta_genetic_tools`, `readers/gff.py::read_gff3` / `write_gff3` (two-pass parent resolution, gzip-aware, multi-parent expansion, `##FASTA` boundary), `readers/bed.py::read_bed`. `FEATURE_QUANT` extended with a `feature_origin` column (vocabulary `'gene_id'` | `'transcript_id'` | `'cluster_id'` | `'genetic_tool'` | `'protein_uniref'` | `'protein_hash'`); `transcriptome.quant.to_feature_quant` projects the existing naive-ORF output into the canonical schema under `feature_origin='protein_hash'`. CLI wired: `constellation reference {import, fetch, search, list, where, link, default, remove, summary, validate}`; `fetch` uses stdlib `urllib.request` against Ensembl / Ensembl Genomes / RefSeq URL conventions (no NCBI Datasets CLI dep); `remove` deletes a cached `(organism, release)` and auto-retargets `defaults.toml` + `current` to the most-recently-fetched survivor when the removed entry was the default/current, mirroring the auto-default-on-first-fetch behaviour so an organism with ≥1 install always has both pointers populated. `Assembly.to_reference` renamed to `to_genome_reference`; companion Annotations build from downstream BUSCO / repeat / telomere passes (still stubbed).

**Mode A genome-guided gene counting shipped (S2):** `constellation transcriptome align --demux-dir <s1-out> --reference <handle> --output-dir <dir> [filter knobs] [--resume] [--progress]` chains directly off the S1 demux output — reads input BAM paths from the typed demux manifest (`transcriptome.manifest.read_manifest_dir`) and loads the resolved `Samples` container from `<demux-dir>/samples/` via `load_samples`; **`--samples` is not a flag on align** (the user supplies it once at demux time, and the persisted ParquetDir bundle rides forward through every downstream stage). `--reference` accepts a reference cache handle (e.g. `homo_sapiens@ensembl-111`); for an off-cache ad-hoc reference root, pass `--reference-dir <PATH>` instead (escape hatch — manifests written this way omit `reference_handle` and can't be opened in the genome browser dashboard until the reference is imported via `constellation reference import`). The output manifest (`manifest.json`) is schema v3: top-level `reference_handle`, `reference_path`, `assembly_accession`, plus `outputs` for downstream consumers (cluster propagates these forward, the viz layer matches sources to references via the handle). Demux now also emits a typed v3 manifest (`kind="demux"`), so the three stages share one `read_manifest_dir` / dataclass-dispatch contract. Three stages: (1) `align` — `map_to_genome` materialises a cached genome FASTA + `.mmi` index from the `GenomeReference`, runs minimap2 splice-aware (`-ax splice -uf --cs=long --secondary=no`) per input file, then `samtools cat | samtools sort | samtools index` into one coordinate-sorted `bam/aligned.bam`. (2) `decode_filter_overlap` — `_bam_alignment_chunks` plans virtual-offset chunks; the fused worker (`fused_decode_filter_overlap_worker`) decodes each chunk via `read_bam_alignments_chunk`, applies `apply_filter_predicates` (length / aligned-fraction / mapq / primary-only — vectorised pa.compute), then `compute_gene_overlap` (sort+searchsorted sweep-join against the broadcast gene set, best-overlap selection, antisense gating). Each worker writes 4 partitioned shards: `alignments/`, `alignment_tags/`, `gene_assignments/`, `stats/`. Worker-local alignment_id packed as `(worker_idx << 32) | local_idx` for collision-free partition concatenation. (3) `count` — `count_reads_per_gene` materialises `gene_assignments/` + `read_demux/` as pa.Tables (bounded ~3 GB / ~10 GB at 200M-read scale), hash-joins on `read_id`, group_by-sums to `FEATURE_QUANT(feature_origin='gene_id')`, and populates `tpm` per-sample as `count × 1e6 / sum_in_sample(count)` — long-read TPM convention (one read = one transcript, no length normalisation). The CLI then writes wide gene × sample TSVs alongside the canonical parquet via `quant.gene_matrix.build_gene_matrix` + `render_gene_matrix_tsv`: `gene_counts.tsv` and `gene_tpm.tsv` (controlled by `--matrix-format {count,tpm,both,none}` default `both`, with `--matrix-min-count N` to drop low-total rows; gene rows carry feature_id / gene_name / gene_biotype / contig / start / end / strand / length annotation columns + one float column per sample, with zero-fill for genes absent from `feature_quant` for stable cross-run indexing). Single output entry point (`pa.Table` inputs) serves both pipeline (CLI materialises from datasets) and tests/Jupyter (callers pass small in-memory tables).

**Streaming pipeline rule:** the in-memory `Alignments` container + `Alignments.filter` (with per-stage `metadata_extras['filter_steps']` audit) + `count_reads_per_gene` API are for tests + Jupyter only — at 30–200M alignments per real run (Pichia ~30M, mouse/Arabidopsis ~200M), the pipeline never materialises a full Alignments. The fused-worker streaming + bounded-resolve-stage memory pattern is the only production path. The same rule extends to **every resolve-stage aggregator** — `build_pileup`, `aggregate_junctions`, `assign_blocks_to_exons`, `compute_exon_psi`, and `cluster_by_fingerprint` — all of which use **Arrow joins / group_by / sort_by for the cardinality-bounded shape operations + PyTorch CPU tensors for the per-partition numerical kernels** (sweep-line RLE, searchsorted interval-overlap, scatter-add deltas, boundary scans). `to_pylist()` followed by Python row loops over alignment-block-cardinality data is **forbidden** on the resolve-stage hot path — it pinned ~12+ hours on a single thread at mouse scale in the pre-refactor code, with hundreds of GB of Python-object growth. `build_pileup` and `assign_blocks_to_exons` take an optional `output_path` arg; when provided they're **streaming Parquet writers** that emit per-partition / per-batch `pa.RecordBatch`es directly to disk without ever holding the aggregate in RAM (`compute_exon_psi` accepts the resulting path back as a `pa.dataset` to extract distinct `(alignment_id, data_exon_id)` pairs in batches). **CLI namespace rule:** gene-counting is a transcriptomics analysis (consumes a reference; not a reference-management op), so it lives at `transcriptome align`, not `reference align`. Mode B (de novo clustering) shares the same verb when it lands — dispatch on `--reference` / `--reference-dir` presence; both absent → "not yet implemented" with exit 2. **Two-layer minimap2 split:** `align/minimap2.py::minimap2_run(target, queries, *, output_path, args, threads)` is the generic, use-case-agnostic subprocess wrapper (knows nothing about splice / preset); `align/map.py::map_to_genome` is the use-case orchestrator that hard-codes the splice flags. Future `map_assembly` / `map_dna_to_genome` / PAF-output verbs all compose `minimap2_run` directly with their own arg lists — no subclassing or wrapping. mappy is rejected (user-confirmed slower); the `map_reads` two-backend stub is deleted. **Memory budget for the resolve stage:** `gene_assignments` ~3 GB + `read_demux` ~10 GB at 200M reads — fits comfortably on workstation RAM. The bigger derived outputs (`alignment_blocks/` ~25 GB, `alignments/` ~30 GB, `block_exon_assignments` ~50 GB on disk at 1.6B-row mouse scale) are now bounded in RAM by the streaming-Parquet handoff between `build_pileup` → `build_derived_annotation` → `compute_exon_psi`. If we cross 1B reads, `count_reads_per_gene` lifts into an Arrow-Dataset-streaming variant — single-function refactor at that point, not a new public API.

**Phase 1 alignment-derived intermediates — now default-on (was opt-in via `--emit-*`):** three core schemas in `schemas/alignment.py` — `ALIGNMENT_BLOCK_TABLE` (one row per CIGAR-derived M/=/X-contiguous block; cs:long-derived `n_match`/`n_mismatch` populated when minimap2 emitted `--cs=long`, null when only CIGAR is available; contig comes via the `alignment_id` FK to `ALIGNMENT_TABLE.ref_name` to save a column × ~600M rows at PromethION scale), `INTRON_TABLE` (single denormalised intron view: one row per *observed* `(contig_id, donor_pos, acceptor_pos, strand)` tuple, augmented with `intron_id` cluster ID + `is_intron_seed` bool — multiple rows can share an `intron_id`; replaces the retired `SPLICE_JUNCTION_TABLE`), `READ_FINGERPRINT_TABLE` (per-read splicing-topology hash via `xxhash.xxh64` over the canonical byte layout `contig_id (int64 LE) || strand (utf-8) || n_introns (uint32 LE) || intron_id_1 (int64 LE) || ...`; primary alignments only; the Phase 2 cluster key — donor/acceptor positions don't enter the hash directly, just the intron_ids assigned by `cluster_junctions`). Producers: `align/cigar.py::parse_cigar_blocks` + `parse_cs_long_blocks` (state-machine parsers; block-break rule is N + D≥`intron_min_bp` but never long I); `quant/genome_count.py::extract_alignment_blocks` (worker-layer orchestrator); `quant/coverage.py::build_pileup` (sweep-line RLE pile-up via per-partition torch `unique` + `scatter_add_` + `cumsum`; per-sample stratification via `read_to_sample: pa.Table` Arrow-join with a `sample_id=-1` sentinel when None; streaming Parquet writer when called with `output_path`); `quant/junctions.py::aggregate_junctions` + `cluster_junctions` (Arrow `sort_by([alignment_id, block_index])` on a NUMERIC-only block projection — strings stay out of the heavy sort + combine_chunks; torch diff on consecutive `alignment_id` to find junction rows; `group_by([contig, donor, acceptor, strand]).count(alignment_id)` after attaching strand/ref_name via two final small joins. `count(alignment_id)` matches `count_distinct(read_id)` since `alignment_id` is 1:1 with `read_id` on primary alignments and a plain `count` is much cheaper than `count_distinct`. Motif lookup prefetches each contig's sequence ONCE into a per-call dict — `GenomeReference.sequence_of` is uncached and decodes the full chromosome string per call, so per-junction calls degenerate catastrophically past ~10K unique junctions); `transcriptome/fingerprints.py::compute_read_fingerprints`. CLI flags on `transcriptome align`: `--no-cluster-junctions` (opt-out for diagnostic raw-only output), `--intron-tolerance-bp INT` (default 5), `--intron-motif-priority STR,STR,...` (default `GT-AG,GC-AG,AT-AC`), `--no-coverage`, `--no-derived-annotation`, `--min-exon-depth INT` (default 5), `--min-intron-read-count INT` (default 3), `--emit-cs-tags` (still opt-in), `--intron-min-bp` (default 25). Wired through the existing fused worker (`emit_blocks=True` always-on) + resolve-stage aggregators that materialise `alignment_blocks/` + `alignments/` once and emit `introns.parquet` / `coverage.parquet` / `derived_annotation/` / `block_exon_assignments.parquet` / `exon_psi.parquet`.

**Part A — greedy support-ranked intron clustering:** `quant/junctions.py::cluster_junctions(introns, *, tolerance_bp=5, motif_priority=("GT-AG", "GC-AG", "AT-AC")) -> pa.Table` replaces the uniform-quantum `(d // q) * q` fingerprint quantisation. Algorithm: per `(contig_id, strand)` partition, sort raw junctions by `(read_count desc, motif_rank asc, contig asc, donor asc, acceptor asc)`; for each junction, bisect-search per-partition seed list for any seed within ±W on BOTH donor AND acceptor (L∞ ball, not L₂); absorb if found, else become a new seed. Output is a single `INTRON_TABLE`-shaped table — same row count as input, with `intron_id` + `is_intron_seed` re-assigned. The window concept is fully absorbed at clustering time; fingerprint hashing then snaps per-read junctions to canonical `intron_id` via hash-join. **Breaking API change:** `compute_read_fingerprints(alignment_blocks, alignments, contigs, introns)` requires the introns table now and ignores the legacy `intron_quantum_bp`. The CLI flag rename is `--intron-quantum-bp` → `--intron-tolerance-bp` on both `transcriptome align` and `transcriptome cluster`. The cluster command's tolerance default is "read from align manifest"; supplying `--intron-tolerance-bp INT` re-runs `cluster_junctions` against the raw per-position rows in `introns.parquet` before fingerprinting, so users can sweep tolerance without re-running align.

**Why support-ranked clustering vs uniform quantisation:** uniform `(d // q) * q` bin edges are arbitrary — a dominant donor at position 1239 hashes into bin 1230 while a stray cryptic at 1240 hashes into bin 1240, splitting reads that biology says agree. The right physical model is asymmetric — one dominant `(donor, acceptor)` per intron with a long tail of low-support cryptics, all anchored to the same GT-AG dinucleotide pair. Support-ranked clustering picks the highest-support junction as the cluster seed (with GT-AG > GC-AG > AT-AC > non-canonical motif tiebreak) and absorbs neighbours within tolerance. The user-facing tolerance is now an explicit `±bp` window, not an implicit binning grid.

**Part B — derived annotation + exon discovery (default-on):** `quant/derived_annotation.py::build_derived_annotation(coverage: pa.Table | Path, introns, alignment_blocks, alignments, contigs, *, read_to_sample: pa.Table | None = None, block_assignments_output_path: Path | None = None, min_exon_depth=5, min_intron_read_count=3) -> (Annotation, BLOCK_EXON_ASSIGNMENT_TABLE | Path, EXON_PSI_TABLE)` derives a data-informed annotation from the always-on Phase 1 substrates. When `block_assignments_output_path` is set, the M:N edge table streams to that Parquet file and `compute_exon_psi` reads it back as a `pa.dataset` to extract distinct `(alignment_id, data_exon_id)` overlap pairs in batches — the table never materialises in RAM. Pipeline: (1) `derive_exons` filters introns to clusters where `group_by(intron_id).sum(read_count) >= min_intron_read_count` (filter on cluster total, not per-row); coalesces RLE coverage above `min_exon_depth`; per `(contig, strand)`, segments each covered interval at trusted intron donor + acceptor positions via `bisect`-slice against a sorted-once-per-partition cut list (an `O(log N + K)` lookup per interval, **not** the `p for p in cuts_set if iv_start < p < iv_end` set-comprehension that pre-refactor pinned a single thread for hours at ~80K cuts × tens of thousands of intervals × 50 partitions); drops pieces whose `(start, end)` exactly matches a trusted intron's span (these are introns, not exons) — keeps the rest as minimal exon segments. (2) `roll_up_genes` builds an exon graph (edge iff a trusted intron seed has `donor==exon_a.end` AND `acceptor==exon_b.start` on the same `(contig, strand)`); connected components via `core.graph.network.Network` are derived genes; each exon's `parent_id` rewrites to its gene's `feature_id`. Output writes to `FEATURE_TABLE` with `source='constellation_derived'`. (3) `assign_blocks_to_exons` produces a long-form M:N edge table `BLOCK_EXON_ASSIGNMENT_TABLE` keyed on `(alignment_id, block_index, data_exon_id)` — most rows 1-to-1 (a clean spliced block falls inside one exon), but boundary-jitter / intron-retention reads emit 2+ rows per block. Per-contig vectorised torch `searchsorted` + `repeat_interleave` expansion with **two-sided bounds** — `hi = searchsorted(exon_starts_sorted, block_end)` for `exon_start < block_end`, AND `lo = searchsorted(cummax(exon_ends_sorted), block_start)` (the monotonic prefix-max of `exon_end` in start-sorted order) for "could this exon possibly still be active at block_start". The lower bound is load-bearing: with `lo=0` the candidate range degenerates to ~all exons per block at scattered-exon scale and explodes intermediate memory by ~10,000×. Adaptive batch size caps peak intermediate at ~250M int64s regardless of `n_exons`. (4) `compute_exon_psi` produces `EXON_PSI_TABLE` with per-(exon, sample) inclusion / exclusion / PSI substrate where `PSI = n_inclusion / (n_inclusion + n_exclusion)` (FLAIR / IsoQuant convention; gene-spanning alignment with any block overlap → inclusion, gene-spanning with no overlap → exclusion). Gene-spanning classification uses per-contig torch chunked broadcasting (auto-tuned ~1 GB intermediate per chunk) to enumerate `(alignment, gene)` overlap pairs. `role_id` is emitted as int8 (0=spanning, 1=partial) directly from `_classify_alignments_per_gene` — not a `"spanning"` / `"partial"` string column — because at PromethION scale the 419M-row aln_exon_role cross-join carries it forward and string vs int8 is a ~4 GB peak difference. The left-join + indicator + group_by chain runs in **streaming chunks via `_join_aggregate_streaming`** (~50M-row chunks; zero-copy `Table.slice`) with per-chunk partial group_by sums merged in a final pass — materialising the full 419M-row left-join would otherwise add ~20 GB transient. Three new schemas: `BLOCK_EXON_ASSIGNMENT_TABLE` in `schemas/alignment.py`, `EXON_PSI_TABLE` in `schemas/quant.py`, plus the `derived_annotation/` ParquetDir bundle (drop-in for `Annotation`'s `ParquetDirReader/Writer`). Opt-outs: `--no-derived-annotation` skips Part B entirely; `--no-coverage` suppresses `coverage.parquet` from final outputs (`build_pileup` still writes it as a transient that `build_derived_annotation` reads back; the CLI unlinks the file after the derived-annotation pass when `--no-coverage` is set). v1 limitations: pure single-exon genes (no introns) are not discovered; coverage stratification is unstranded (per-strand exon derivation uses unstranded coverage as a depth-thresholding substrate); differential exon-abundance estimation (negbin LRT across samples) is deferred — `EXON_PSI_TABLE` is the v2 substrate.

**Phase 2 genome-guided fingerprint clustering shipped:** `constellation transcriptome cluster --mode genome-guided --align-dir <s2-out> --demux-dir <s1-out> --output-dir <dir> [--build-consensus] [--max-5p-drift 25] [--max-3p-drift 75] [--intron-tolerance-bp INT] [--min-cluster-size 1] [--per-sample-clusters] [--write-fasta] [--write-summary] [--threads N] [--resume] [--progress]` ships an isoform-resolution clusterer that consumes the always-on `alignment_blocks/` + `introns.parquet` from align. `Samples` loads from `<demux-dir>/samples/` (no `--samples` flag); the align manifest is read via the typed `read_manifest_dir`. Algorithm: (1) load `introns.parquet`; if `--intron-tolerance-bp` differs from the align manifest's value, re-run `cluster_junctions` against the raw per-position rows in-place (sweeping tolerance never requires re-running align); (2) re-derive fingerprints via `compute_read_fingerprints(alignment_blocks, alignments, contigs, introns)` — Arrow-native pipeline: sort blocks by `(alignment_id, block_index)` numeric-only, derive per-read junctions via numpy shift (`donor = ends[:-1], acceptor = starts[1:]` masked by same-alignment), hash-join junctions ⨝ introns for intron_id substitution, group-by alignment_id, then a `xxh64` loop bounded by N_alignments (~3 min @ 200M reads vs blake2b's ~33 min, with zero `to_pylist` on block cardinality); (3) `cluster_by_fingerprint` resolves per-fingerprint row indices into each source table once via `pc.index_in` (reads, primary_lookup, metrics), sorts a narrow all-numeric keys table `(row_idx, contig_id, strand, sample_id, fingerprint_hash, reads_idx, primary_idx, metrics_idx?)`, walks cluster boundaries on the narrow sort, and `pc.take`s per-cluster from each source — the wide `enriched` table that previously dragged `sequence` (~50 GB at PromethION scale) through the global sort and crashed with `ArrowInvalid: offset overflow while concatenating arrays` is gone; (4) strand-aware 5'/3' span-coherence drift filter — on `+`, 5' = `span_start`; on `-`, 5' = `span_end`; singletons skip the filter; (5) Layer-0 dereplicate trimmed transcript-window sequences (blake2b-8 hash of `READ_TABLE.sequence[transcript_start:transcript_end]`) per cluster and pick the most-abundant unique sequence's read as representative (tiebreak by mean `dorado_quality`); (6) optional `--build-consensus` per-position weighted-PWM majority vote on the genome coordinate frame via the shared `align/consensus.py` kernel — `alignment_cs` accepts either `pa.Table` (eager, for tests + small Jupyter) or `pa.dataset.Dataset` (the CLI default; per-cluster filter pushdown materialises only the ~tens of cs:long rows for the current cluster, sidestepping the ~50 GB eager load). **Mega-cluster streaming refactor:** `_emit_cluster` now operates entirely on Arrow columns + numpy without ever calling `slice_table.to_pylist()` — drift filter is vectorized numpy on `span_start`/`span_end`, Layer-0 derep is `pa.Table.group_by("sequence").aggregate(...)` (collapsing 2.15M-row mega-clusters to ~100K-500K unique-sequence groups in C++-speed), membership rows are built via `pa.Table.from_arrays(...)` with `pc.if_else` for the per-row role/metric branches, and `_emit_cluster` returns `(cluster_row_dict, membership_table: pa.Table)` so the per-cluster loop accumulates a `list[pa.Table]` and `pa.concat_tables(...)` at the end (zero-copy when schemas match) instead of materialising tens of GB of Python dicts. `build_consensus` adds `threads: int = 1` + `scatter_chunk_size: int = 100_000` kwargs: per-member `_cs_votes` parsing dispatches across a `ThreadPoolExecutor` (regex releases the GIL) and per-chunk results accumulate into ONE batched `flat_pwm.index_add_(...)` per ~100K members instead of one scatter call per member. CLI `--threads N` forwards to `consensus_threads` through `cluster_by_fingerprint` so the mega-cluster consensus drops from ~30-60 min sequential to ~2-5 min with 16 threads. Two schemas in `schemas/transcriptome.py`: `TRANSCRIPT_CLUSTER_TABLE` with `mode` ∈ `{'genome-guided', 'de-novo'}` discriminator + `contig_id`/`strand`/`span_start/end`/`fingerprint_hash`/`n_unique_sequences`/`sample_id` and `CLUSTER_MEMBERSHIP_TABLE` (long-form (cluster_id, read_id) edges with `role` ∈ `{'representative', 'member', 'duplicate', 'drift_filtered'}` + multi-parameter fit-to-cluster scores: signed `drift_5p_bp` / `drift_3p_bp` from cluster median + cs:long-derived `match_rate` / `indel_rate` / `n_aligned_bp`). Phase 1 retrofit: `extract_alignment_cs(alignments, tags) -> ALIGNMENT_CS_TABLE` + the `--emit-cs-tags` flag on `transcriptome align` writes a sixth `alignment_cs/` partitioned shard preserving per-alignment cs:long strings (still opt-in — required by `--build-consensus` to skip the prohibitively-slow re-run-NW-per-member alternative). Drift-filtered reads are *kept* in `cluster_membership` with `role='drift_filtered'` by default (Phase 4 cross-validation); `--drop-drift-filtered` opts out. Cluster-pair merge for subset-junction-noise is deferred to v2.

**Granularity contract for `transcriptome cluster --mode genome-guided`:** clustering is **per-isoform**, not per-gene. The `fingerprint_hash` now keys on `(contig_id, strand, [intron_id_1, intron_id_2, ...])` (no donor/acceptor positions enter the hash directly — the intron_id is the canonical cluster identifier assigned by `cluster_junctions`). Different splicing topologies on the same gene fall into different clusters; reads agreeing on canonical-intron-id splicing collapse into one cluster regardless of substitution noise (within-cluster heterogeneity surfaces via `n_unique_sequences > 1` and per-membership `match_rate` < 1.0). A gene with N annotated isoforms produces ≥ N clusters in v1; the inequality picks up novel isoforms (correctly distinct), truncated reads that miss an internal junction (incorrectly split; deferred subset-junction merge is the v2 fix), and basecaller-induced junction-call errors (rare; surfaces as singletons). Alt-TSS / alt-polyA isoforms whose intron chain is identical fall into the *same* fingerprint cluster initially and the drift filter separates the long-tail divergent reads.

**Default-output disk impact:** PromethION-scale runs (~600M reads, ~600M blocks) produce `alignment_blocks/` ≈ several GB, `introns.parquet` ≈ tens of MB, `coverage.parquet` ≈ 100 MB-1 GB, `derived_annotation/` ≈ tens of MB, `block_exon_assignments.parquet` ≈ several GB (cardinality matches blocks), `exon_psi.parquet` ≈ tens of MB. Workstation-scale runs (~10⁷ reads) all stay sub-GB. Footprint-sensitive users opt out of derived annotation via `--no-derived-annotation` (saves the largest secondary output, `block_exon_assignments.parquet`); `--no-coverage` suppresses `coverage.parquet`; `--no-cluster-junctions` retains the raw per-position view in `introns.parquet`.

**Phase 3 de novo first-round assembly shipped (`cluster/denovo/` subpackage):** `constellation transcriptome cluster --mode de-novo --demux-dir <s1-out> --output-dir <dir> [--identity 0.98] [--max-5p-overhang 30] [--max-3p-overhang 30] [--kmer 15] [--window 10] [--min-cluster-size 1] [--min-abundance 1] [--max-cluster-rounds 1] [--error-model {default,empirical}] [--overdispersion 0.0] [--[no-]predict-orfs] [--min-aa-length 60] [--emit-cluster-detail] [--detail-top-n 50] [--threads N] [--[no-]report] [--resume] [--progress]` is a **reference-free** clusterer over the Complete, non-fragment transcript windows (loaded via `align/map.py::_iter_demux_read_batches(only_complete=True)` — same substrate as the demux protein groupings). Pipeline: `dereplicate` (Arrow `group_by("sequence")` → unique seqs + abundance + read_map) → `encode`+`minimizers` (ASCII→uint8 `torch.frombuffer`, canonical k-mer SplitMix64 rolling hash, window-min via boundary-masked `unfold().min`, chunked at sequence boundaries so the per-position hash intermediate stays bounded) → `candidates` (sorted-index anchor-star with diagonal-consistency + high-frequency stop-word cap — note the cap floor is high so small/expression-concentrated datasets aren't over-pruned, the star is already linear in bucket size, verify supplies precision) → `verify` (edlib HW, **shorter→longer**, accept iff identity ≥ `--identity` AND 5'/3' overhang ≤ thresholds; emits + caches the CIGAR — `task=path` is ~free, so **edlib runs once per edge**, reused by consensus) → `cluster_graph` (**scipy connected-components**, NOT radius-1 greedy: at ~1% error almost every read is a distinct unique with abundance≈1, so the roadmap's abundance-anchored greedy under-clusters; CC over the 98%-verified graph collapses all minor variants while distinct transcripts (>2% divergent) share no edge; centroid = highest-abundance/longest member) → `consensus` (centroid-anchored CIGAR-walk PWM via `torch.index_add_`, reuses cached CIGARs, re-aligns only the component-path members not directly verified to the centroid) → ORF on consensus (reuses `core.sequence.nucleic.find_orfs`, ATG-only parity table) → `quant` (`FEATURE_QUANT(feature_origin='cluster_id', engine='constellation_denovo')` + per-sample TPM). **Variant characterization (the "what was collapsed" deliverable):** `variants.py` calls within-cluster variants with a **context-aware, depth-gated error model** — the binomial (or beta-binomial when `--overdispersion>0`) upper-tail SF under a context-conditional null ε (homopolymer-indel by run length / substitution / non-hp-indel; explicit priors in `ErrorModel`, the hifiasm non-uniform-error insight), BH-FDR per cluster, `n < n_min → ambiguous`; the binomial `n` is raw read multiplicity (= the abundance-weighted PWM column sums). `--error-model empirical` re-fits ε from cluster-wide high-confidence disagreements (high depth, major fraction >0.95) and re-tests every variant position with no second consensus pass. `haplotypes.py` builds the read×variant-position allele matrix from the retained PWM votes, collapses to distinct allele-combination haplotypes (abundance-weighted), and computes pairwise **r²** (phasing) → `max_linkage_r2` per variant. Outputs: `clusters.parquet` (TRANSCRIPT_CLUSTER_TABLE, `mode='de-novo'`, genome cols null, consensus + ORF populated), `cluster_membership.parquet`, `cluster_variants.parquet` + `cluster_haplotypes.parquet` (new schemas in `cluster/denovo/schemas.py`), `cluster.fa` + `proteins.fasta` + `feature_quant.parquet` + `cluster_counts.tsv`, `diagnostics/report.md` (default-on aggregate), and — under **`--emit-cluster-detail` (off by default; the verbose per-cluster printouts)** — per-cluster `detail/cluster_<id>/{haplotype_map.svg, variants.tsv}` (the MSA-free **haplotype heatmap**: rows = haplotypes, cols = variant positions, cells = allele with consensus muted / minor saturated). The cluster manifest writes `align_dir=""` (no align dir; reference fields null). **Threshold note:** `--identity` is a read-to-read (pairwise) gate; two reads each ~1% from the true transcript are ~2% apart, so for ~1% error reads ~0.96-0.97 reduces low-abundance-tail fragmentation while the deep clusters are intact at 0.98 — `max_cluster_rounds`>1 (consensus-anchored re-clustering, the principled fix) is reserved. Parallelism: minimizers/candidates/PWM are torch-vectorized; verify + per-cluster consensus fan out via fork-based `ProcessPoolExecutor` with copy-on-write-shared sequence lists (no per-task sequence pickling).

**Still stubs:** `Assembly` / `Project` container method bodies, `Dorado` runner, the legacy `transcriptome/cluster/cluster.py` / `cluster/consensus.py` Phase-6 mmseqs stubs (superseded by `cluster/denovo/`), `--mode validate` (Phase 4 cross-validation) branch of `transcriptome cluster`, assembly / polish / scaffold / annotation runners. **Minimap2 splice-mode preset system (`align/presets.py` + `data/sequencing/minimap2_splice_presets.json`).** The splice DP default `-G 200000` invites adjacent-gene-fusion artifacts on compact / dense genomes. Three v1 organism-class profiles ship in the JSON bundle and compose on top of the hardcoded splice base flags (`-ax splice -uf --cs=long --secondary=no`) via `resolve_minimap2_args(base_args=..., profile=..., max_intron_length=..., non_canonical_cost=..., junc_bed=..., junc_bonus=..., extra_args=...)`:

| Profile | -G (max intron) | -C (non-canonical cost) | Examples |
|---|---|---|---|
| `compact_eukaryote` | 5,000 | 5 | Pichia, S. cerevisiae, Aspergillus |
| `intermediate_eukaryote` | 50,000 | 5 | Drosophila, C. elegans, Arabidopsis |
| `animal` | 200,000 | 5 | Human, mouse, zebrafish |

Resolution order (later wins for the same flag): base_args → preset → explicit overrides → extra_args. Conflicting `extra_args` against a kwarg-managed flag (-G, -C, --junc-bed, --junc-bonus) raise `ValueError` to refuse silent overrides. CLI wiring: `transcriptome align` exposes `--organism-profile {compact_eukaryote,intermediate_eukaryote,animal}` (default None → just base flags + stderr warning that minimap2's stock 200 kb cap is in effect), `--max-intron-length`, `--non-canonical-cost`, `--junc-bed`, `--junc-bonus`, `--minimap2-extra "string"` (shlex-split escape hatch). The align manifest gains a `minimap2_resolved_args: list[str] | None` field (additive, default-None — pre-existing v4 manifests load cleanly without it) recording the exact arg tuple minimap2 was invoked with. Adding a new preset: append an entry to `scripts/build-minimap2-splice-presets-json.py` (hand-curated Python literal), regenerate the JSON, ship.

**Diagnostic-report system (`transcriptome/{align,cluster}/diagnostics.py` + `transcriptome/_render.py`).** Each pipeline stage emits a markdown report with embedded SVG figures under `<output>/diagnostics/`. Metrics are pure functions over the existing parquet outputs (no extra pipeline state); reports are auto-emitted at the end of `transcriptome align` / `transcriptome cluster` (opt out with `--no-report`) and can be regenerated read-only via `constellation transcriptome diagnose --align-dir DIR [--cluster-dir DIR] [--reference HANDLE] [--organism-profile PROFILE]`.

- **Align report** (`<align-dir>/diagnostics/report.md`): run parameters table + minimap2 resolved-args, intron length distribution colored by motif, splice motif composition (GT-AG / GC-AG / AT-AC / other by both cluster count and read count), annotated junction agreement by supporting-read tier, alignment complexity (blocks per primary), MAPQ distribution, **multi-gene alignment counts** (the fusion smoking gun — per-alignment count of overlapping annotated genes, flagged at >2% multi-gene).
- **Cluster report** (`<cluster-dir>/diagnostics/report.md`): run parameters, cluster size distribution (singleton-fraction + top-cluster flags), cluster span vs annotated gene length, **multi-gene cluster overlap** (same fusion smoking gun on the cluster stage), drift filter statistics, layer-0 dereplication ratio, intron-chain complexity (joins back to upstream align dir to count representative-alignment blocks; flags mega intronless buckets), consensus quality (per-member match_rate / indel_rate).

Flag thresholds bubble into a top-of-report **FLAGS** section so a reader skimming the report sees red flags first. Per-metric failures are caught by the orchestrator's `_safe` wrapper and render as a stub note — a single broken metric never breaks the whole report, and report-generation failures during the auto-emit never break a successful pipeline run (the CLI logs a warning + a re-run hint pointing at `constellation transcriptome diagnose`).

Figures: SVG via matplotlib Agg backend (lazy-imported inside `_render.save_svg_figure`). Renders inline in GitHub / VSCode markdown previews; small file size for the chart types we emit (histograms, bar charts, small overlays). Flag thresholds for the intron-length section are profile-aware (`_PROFILE_MAX_BIOLOGICAL_INTRON_BP`: 50 kb / 100 kb / 1 Mb for compact / intermediate / animal). Adding a new metric: write a pure function `section_<name>(table, *, figures_dir, ...) -> ReportSection`, register it in the per-stage orchestrator's `sections.append(_safe(...))` chain; the FLAGS hoist is automatic. The standalone `diagnose` verb picks the new metric up for free.

## Conventions (sequencing layer)

- **Sequencing containers do NOT mirror the massspec Library/Quant/Search trio.** Sequencing's natural shape is `acquisitions + samples + reads + alignments + (genome reference, annotation, transcript reference) + assembly + quant + annotation tools + genetic_tools`. The MS theoretical/empirical split doesn't transfer because for de novo work the "library" *is* the experimental output. The reference layer is **split into three sibling Library-shaped containers** because real-world references are heterogeneous: `GenomeReference` (`CONTIG_TABLE` + `SEQUENCE_TABLE`), `Annotation` (`FEATURE_TABLE`, FK into a `GenomeReference`), `TranscriptReference` (`TRANSCRIPT_TABLE`; populated either from a paired GenomeReference + Annotation via `TranscriptReference.from_annotation` or from a standalone transcript FASTA when no genome exists). Each ships PK/FK closure + ParquetDir native form + Reader/Writer Protocols. `GeneticTools` is the cRAP-equivalent for genomics — a curated bundle of common cloning / engineering parts (AbR / FPs / tags / promoters / common enzymes / vector backbones) at `constellation/data/genetic_tools.json`. `Assembly` is its own container with provenance Reference doesn't need (read-coverage, polishing rounds, scaffolding edges) and `Assembly.to_genome_reference()` lifts a finalized assembly into the same `GenomeReference` shape so external-ref and de-novo-ref workflows share downstream code; companion Annotations build separately from BUSCO / RepeatMasker / telomere passes. `Quant` keys per-feature counts/coverage/TPM on `(feature_id, sample_id, engine, feature_origin)` — multi-tag, see below. `Alignments` is its peer container (per-acquisition, FK to `GenomeReference`). No `Search` analog — alignments are observations, not detection scores.

- **`feature_quant` is multi-tag, not strict-precedence.** `FEATURE_QUANT` carries a `feature_origin` column with vocabulary `'gene_id'` | `'transcript_id'` | `'cluster_id'` | `'genetic_tool'` | `'protein_uniref'` | `'protein_hash'` (frozen set in `sequencing.schemas.quant.FEATURE_ORIGIN_VOCAB`). `feature_id` is unique within a `feature_origin` partition; sums within a partition are valid counts; **never sum across partitions** (different statistical models of "what a feature measures"). The transcriptome S1 naive-ORF output projects into FEATURE_QUANT under `feature_origin='protein_hash'` via `transcriptome.quant.to_feature_quant`; future genome-aligned gene counts use `'gene_id'`; transcriptome-mode EM lands `'transcript_id'`; mmseqs2 hits land `'genetic_tool'` or `'protein_uniref'`.

- **External-format readers are hand-rolled, stdlib only.** `readers/fastx.py::read_fasta_genome` / `read_fasta_transcriptome` / `read_fasta_genetic_tools`, `readers/gff.py::read_gff3` / `write_gff3`, and `readers/bed.py::read_bed` are pure-Python streaming parsers (gzip-aware via stdlib). No `pyranges`, no `gffutils`, no pandas. Reasoning: pyranges1 pulls pandas + a gtfreader sub-dep + a Rust backend; pysam.FastaFile only earns its keep on huge genomes that need `.fai`-indexed random contig access (we do full one-pass reads). Same precedent as `readers/sam_bam.py` — pysam stays scoped to where its CIGAR / virtual-offsets / sorting machinery is genuinely irreplaceable. Coordinates are 0-based half-open internally; GFF3 (1-based inclusive) and BED (already 0-based half-open) convert at the reader boundary.
- **`Samples` is first-class in sequencing, not derived.** A sample = `(acquisition_ids, barcode_id?)` resolved via the M:N `SAMPLE_ACQUISITION_EDGE` table. Single-sample multi-flowcell (genomic ultra-HMW; barcode null) and multi-sample single-flowcell (multiplexed transcriptomic; M barcodes) and mixed designs all use the same schema — no bifurcation by experiment type.
- **Generic `ragged_trace_1d` schema covers POD5 + future MS scan blobs.** Variable-length 1D-array-per-row with rich row-level metadata earns a slot in `core.io.schemas` (≥2 modalities). POD5 reads are 1k-5M+ samples each (~10¹³ rows if flattened to `Trace1D`); MS scans are 10⁴-10⁵ peaks each. The factory takes a value dtype + optional explicit-x column + per-row extras: POD5's `RAW_SIGNAL_TABLE` instantiates with `value_dtype=int16, value_name='signal'`, future massspec scans with `value_dtype=float64, value_name='intensity', x_name='mz'`. `ragged_to_tensors` projects the ragged columns to per-row torch tensors (each row may have a different length — caller pads / batches).
- **Demux output is segment-row-shaped, not read-row-shaped.** `READ_SEGMENT_TABLE` carries one row per *located segment* (adapter / polyA / barcode / transcript / UMI) — chimeric / concatemer reads (very common in nanopore) emerge as multiple segment-runs per read without any chimera-aware special casing. Demux logic operates on `LibraryDesign(layout: tuple[Segment, ...])` composed of `AdapterSlot` / `BarcodeSlot` / `PolyASlot` / `UMISlot` / `TranscriptSlot`; new ONT/SMARTer-derived chemistries land as new designs without code changes. `Adapter` and `Barcode` are independent registry-grade objects (kits share adapters across products). `READ_DEMUX_TABLE` is the derived view: one row per resolved transcript window with `sample_id` assigned and `is_chimera` flagged.
- **Sequencing demux uses segmented edlib over Dorado's full-primer Smith-Waterman.** Three-step algorithm (lifted from NanoporeAnalysis): (1) anchor on polyA via error-tolerant 'AAAA' edit-distance scan, (2) locate 3' PCR adapter on polyA-side; score the gap region against barcode panel, (3) locate 5' adapter and isolate transcript window. Beats Dorado's built-in demux because high homopolymer error in nanopore A-runs (lab measurement: mean A=26 against expected A=20) distorts SW scoring of the full primer enough to misassign barcodes; segmenting on polyA first and scoring only the discriminating region sidesteps this.
- **`Scorer` Protocol — pluggable per-segment scoring; `HardThresholdScorer` is production.** `transcriptome.scoring.Scorer` is the seam between hard-threshold and probabilistic scoring of segments (polyA / 5' adapter / 3' adapter / barcode panel). The orchestrator (`transcriptome.demux.locate_segments`) calls `Scorer` methods uniformly — swapping modes is a one-line change at the CLI / API surface, not a fork in the demux algorithm. `BarcodeVerdict.delta_edit_distance` is the gap between the best and second-best matches across the FULL panel (not just within-budget candidates), so `READ_SEGMENT_TABLE.score_delta` is directly useful as a diagnostic — a 0-vs-1 best-second gap is much weaker evidence than a 0-vs-6 gap even though both look like "Complete" matches under the hard threshold. **`ProbabilisticScorer` is deferred indefinitely** per the scoring-domain rule below; the seam stays in place at zero cost so a future basecaller emitting trustworthy posteriors can drop in without re-architecting demux.

- **Scoring-domain rule: don't dissect basecaller artifacts in the demux scorer.** `HardThresholdScorer` matches NanoporeAnalysis `_fixed1` semantics with three deliberate accuracy improvements over the upstream baseline and is the production scorer. The originally-planned `ProbabilisticScorer` (S2) was meant to fit calibrated `Parametric` distributions over polyA length / adapter edit distance / barcode panel scores, with priors learned from the empirical distributions in `analyze-demux-statistics.py`. **We deferred this work after the polyA-by-barcode diagnostic showed that per-barcode polyA-length systematics are dominant and motor-coupled** — see `data/sequencing/cdna_wilburn_v1.json::calibration_notes` (BC06 example: TAT-repeat structure interacts with the motor protein during the polyA dwell, biasing the basecaller's dwell-time → A-count conversion). Calibrating a probabilistic demux scorer to the basecaller's per-barcode artifacts would be curve-fitting to a black box rather than recovering signal. The principled path lives one layer earlier: (a) **signal-level demux** working on the raw POD5 squiggle directly (change-point detection on dwell-time series, not sequence similarity on basecalled output — same domain as `signal.models.physical` for direct-RNA modifications), or (b) **better basecaller training** with constellation supplying the trustworthy-ground-truth fixtures (synthetic-read corpus + lab-measured systematics) a training run needs. The `Scorer` Protocol seam stays — drop-in for whichever path matures first — but no `Parametric` calibration work happens against current Dorado output.

- **The simulator + stats infrastructure is a diagnostic tool, not a calibration substrate.** `transcriptome.simulator` and `scripts/analyze-demux-statistics.py` were originally designed to feed S2's prior calibration. After the scoring-domain pivot above, they keep their value — for spotting basecaller drift across Dorado versions (re-run the stats script on a new Dorado release; if BC06's polyA mean shifts, the basecaller's homopolymer model changed), regression-testing `Scorer` implementations against the synthetic ground truth, and characterising per-(library design × basecaller) systematics. They are NOT a substrate for fitting empirical priors that pretend to recover ground truth from current Dorado output.
- **HardThresholdScorer corrects three NanoporeAnalysis bugs deliberately.** (1) Filter edlib's `editDistance=-1` no-match sentinel before picking a winning barcode (NanoporeAnalysis's filter checks `is not None`, which leaks `-1`s and assigns no-match reads to whichever barcode is first in dict-iteration order). (2) Short-circuit on empty `ref` strings in `locate_substring` — edlib on an empty subject returns `editDistance=len(query)` with `locations=[(None, -1)]`, which NanoporeAnalysis treats as a "match" and ends up assigning bogus 16-edit-distance "matches" against empty putative_umi regions. (3) Pick the non-Unknown orientation cleanly when exactly one orientation classifies as Unknown — NanoporeAnalysis's `cDNA[1] == 'Unknown'` literal-string check fails when reverse is `'Unknown Fragment'` (not just `'Unknown'`) and silently throws away forward's real annotation. All three are documented in code where the divergence happens; constellation's behavior is strictly more accurate than the upstream baseline.
- **PolyA-barcode boundary ambiguity is a scoring concept, not a panel-data property.** When a barcode's RC begins with 'A' (BC05/06/11/12 in `cdna_wilburn_v1`), polyA detection greedily absorbs the leading A — the barcode matches the panel at ed=1 instead of ed=0. The leading A is information-theoretically null under hard-mode scoring (no upper bound on polyA length means no signal in length variation). The Pichia-flowcell diagnostic (calibration_notes in `cdna_wilburn_v1.json`) checked whether stratifying polyA length by leading-A-RC-class recovers a clean bimodal structure that a probabilistic scorer could exploit — answer was **no** (means within ~0.1 nt of each other; per-barcode systematics like BC06's TAT-repeat motor effect are larger and unrelated to leading-A class). The boundary-absorption hypothesis is documented but does NOT appear to be the dominant source of polyA-length variation. Per the scoring-domain rule above, never strip from the panel + never trim at scorer setup — the per-barcode polyA-length empirical distribution stays intact as a Dorado-behaviour diagnostic, and downstream consumers that care about boundary ambiguity address it at the basecaller level (signal-domain demux), not the demux scorer.
- **S1 stage parallelism: per-worker file chunks, fused ingest+demux, partitioned-parquet shards.** `sequencing.parallel.run_batched` is the generic batch executor: each worker writes its own `<output>/<key>/part-NNNNN.parquet` shard directly (no Arrow tables pickled back to the parent), `_SUCCESS` markers gate stage-level resume, missing shards inside an unfinished stage trigger shard-level partial recovery. `transcriptome.stages.run_demux_pipeline` orchestrates `_plan_chunks` (parent-only; cheap chunk-spec computation) → `run_fused_demux_stage` (N parallel workers each do ingest + demux + ORF) → `resolve` (sequential aggregation). For BAM, the planner does a fast no-decode pysam scan (`_bam_record_chunks` in `readers/sam_bam.py`) recording record-boundary virtual offsets every `chunk_size` primary records; for SAM, byte-range chunks split at newlines via the worker-side handshake protocol (each worker discards a partial first line if not at file start, then reads complete lines until past `byte_end`). Each worker opens its own pysam handle on the source file, so ingest is fully N-way parallel — no parent-serial bottleneck. Replaces NanoporeAnalysis transcriptome_nf (Nextflow per-batch fan-out → physical SAM shards on disk) with `--threads N` + `--batch-size` + `--resume` CLI flags. Worker function (`transcriptome.stages._fused_chunk_worker`) lives at module top level so `ProcessPoolExecutor` can pickle it; designs are loaded by NAME inside workers (cheap JSON read; cached) rather than pickled across the IPC boundary. Each worker emits 4 shards: `reads`, `read_segments`, `read_demux`, `orfs` — the `reads` dataset is partitioned (one parquet per worker), queryable via `pa.dataset.dataset(reads/)`.
- **Progress reporting is a Protocol, not stdout calls.** `core.progress.ProgressCallback` accepts `ProgressEvent` (kind: `stage_start` / `stage_progress` / `stage_done`, plus `stage`, `completed`, `total`, `message`, free-form `payload`). CLI binds `StreamProgress` (one line per event to stderr) under `--progress`; future GUI binds the same Protocol to a Qt signal or websocket. The core verbs in `parallel.run_batched` and `transcriptome.stages` take `progress_cb: ProgressCallback | None` and emit at well-known points without caring how the host renders them. The Protocol lives in `core/` rather than `sequencing/` so massspec verbs (the Thermo converter, future mzML / Bruker / Sciex readers) can share it — domain modules never cross-import.

- **Resolve-stage pitfalls observed at mouse / PromethION scale.** Four distinct bottleneck classes were learned the hard way during the resolve-stage rewrite — all single-thread, all looked innocuous in the toy-scale tests, all were measured in hours-to-days of wall time before the fix. Avoid these patterns in any future resolve-stage code:
  1. **Calling a method that does work-per-call inside a hot row loop.** `GenomeReference.sequence_of(contig_id)` does a `ChunkedArray.to_pylist()` over the contig table and then `.as_py()` to decode the entire chromosome sequence (~200 MB for mouse chr1) into a fresh Python str. Innocuous at "tens of thousands of junctions" — catastrophic at 1M+ junctions (the actual mouse number). Always prefetch per-contig values into a dict before any junction-row loop. Same principle applies to any uncached property of `Annotation`, `TranscriptReference`, etc.
  2. **One-sided `searchsorted` bounds on interval-overlap joins.** Both halves of the half-open overlap predicate (`exon_start < block_end` AND `exon_end > block_start`) must contribute to the candidate slice. Sorting by `exon_start` gives only the upper bound; the lower bound needs `searchsorted` against `cummax(exon_end_in_start_sorted_order)` — the prefix-max is monotonic, so binary-searchable, and prunes "exons that have already ended" from the candidate range. Without it, `total = sum(counts) ≈ n_blocks × n_exons` and the intermediate explodes.
  3. **`O(N×M)` Python set scans inside per-partition loops.** `[p for p in cuts_set if iv_start < p < iv_end]` repeated across thousands of intervals × dozens of partitions is the same algorithmic mistake as #2 dressed differently. Sort the set once per partition; `bisect.bisect_right` + `bisect.bisect_left` to slice the interior range.
  4. **String columns in wide-table sort + combine_chunks.** Arrow's string-array `combine_chunks` offsets-rewrite is partially single-threaded; at 1.4B+ rows of `read_id` it pins one core for hours. Project to numeric-only columns BEFORE the sort, attach strings via a final small join on the post-aggregation table. Same caution applies to materialising large datasets via `pa.dataset.to_table()` — pass `columns=[...]` to project at scan time and drop the bulky strings (`cigar_string`, `read_group`, etc.) the resolve stage doesn't need. `compute_exon_psi`'s `role` column is emitted as `int8 role_id` for the same reason: at 419M aln_exon_role rows, "spanning"/"partial" strings cost ~4 GB vs ~400 MB for the int8 encoding.

  A related streaming pattern that's now standard: `assign_blocks_to_exons` and `build_pileup` are streaming Parquet writers when given `output_path`; `compute_exon_psi` consumes `block_assignments` from disk as a `pa.dataset` and stream-extracts distinct overlap pairs; the join-aggregate step inside `compute_exon_psi` uses `_join_aggregate_streaming` to slice `aln_exon_role` in zero-copy chunks rather than materialise the full 419M-row left-join. The combined effect on the production mouse run: resolve stage went from 12+ hours hung (with ~95 GB peak before OOM was even reachable) to ~7-8 min within the 19.5-min end-to-end target, with peak RSS bounded by the alignments + blocks + read_demux materialisation budget.
- **`align/` modules named by verb, not backend.** `pairwise_align(query, ref, *, backend='edlib'|'parasail')`, `map_reads(reads, reference, *, backend='mappy'|'minimap2')`, `locate_substring(query, ref, *, max_distance)`. Backend choice is a kwarg, not visible at the import level — keeps demux / polish / scaffold callers clean and lets us swap implementations without touching call sites.
- **`DoradoModel` parses lab shorthand.** `DoradoModel.parse("sup@v5.0+5mC,5hmC")` → `DoradoModel(family='sup', version='v5.0', chemistry='dna_r10.4.1_e8.2', mods=('5mC', '5hmC'))`. Default chemistry is DNA R10.4.1; explicit chemistry prefix (`"rna004_130bps_sup@v5.0.0+m6A_DRACH"`) handles direct-RNA. `model_name()` renders to Dorado's canonical on-disk model directory name; `modified_bases_arg()` formats for the `--modified-bases` CLI flag.
- **`DoradoRunner` returns `RunHandle`, never blocks.** Single-flow-cell sup-model basecalling takes 3-4 days on an A100 — long enough that blocking the Python session is wrong. `RunHandle` exposes `wait()` / `poll()` / `tail_progress()` / `cancel()`, and `RunHandle.attach(pid_file)` rebinds to a previously-detached run from a different Python session (so `constellation basecall --follow` can monitor a job started days earlier).
- **Pure-functional first; `Project` is the orchestrator on top.** Verbs like `bam_dir_to_assembly(bam_paths)` work standalone with no project context. `Project.open(root)` reads a `manifest.toml`, knows the standard layout (`pod5/ → bam/ → assembly/ → annotation/ → transcriptome/`), and is the *only* layer that mutates filesystem state — same pattern as `Library` / `save_library`. Labs that produce BAMs externally and just want the analysis layer skip the Project entirely.
- **Cross-modality workflows live as top-level scripts (one workflow today).** `constellation/transcriptome_to_proteome.py` is the only such workflow today, so it lives at top-level per the "≥2 workflows before a folder" rule. When a second cross-modality pipeline lands (genome → spectral library, structure → spectral prediction), they co-locate into a dedicated subpackage; name TBD (`bridges/` and `pipelines/` are ruled out — candidates: `asterism/`, `confluence/`, `weft/`, `synthesis/`, `composer/`, `lattice/` — defer until two workflows exist to compare).
