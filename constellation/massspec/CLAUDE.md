# CLAUDE.md — `constellation.massspec`

This file extends the project-wide rules in [../../CLAUDE.md](../../CLAUDE.md). Read that first; chemistry / ProForma / cleavage rules live in [../core/CLAUDE.md](../core/CLAUDE.md).

`massspec/` is the mass-spectrometry domain module. It absorbs Cartographer (the lab's MS proteomics package — first port target) and surfaces Constellation's HUPO-PSI compliance commitments. Layout: a theoretical/empirical/scoring sibling trio (`library/` + `quant/` + `search/`) sharing per-acquisition provenance (`acquisitions/`), peptide-level chemistry (`peptide/`), HUPO-PSI annotations (`annotation/`), cross-tier file-format adapters (`io/`), and per-model tokenizers (`tokenize/`). MS readers (mzML, mzPeak, vendor formats) land here when each format-specific reader ships.

## Code-level invariants (massspec layer)

- **EncyclopeDIA modseq parsing at the reader boundary.** EncyclopeDIA's `[+N.NNN]` mass-delta notation is ambiguous between N-terminal α-amine and residue side-chain placement (e.g. `K[+42.011]VERPD` could be K side-chain Acetyl OR N-terminal Acetyl on K — these are chemically distinct molecules). `constellation.massspec.io.encyclopedia.parse_encyclopedia_modseq` resolves this by looking up each mass delta in UNIMOD and applying the modification's `specificities` constraints: terminal-only mods at residue 0 promote to `[Mod]-X` form; mods that fit both placements default to side-chain unless localization evidence overrides. Mass deltas that don't resolve unambiguously pass through as ProForma 2.0 mass-delta `ModRef` entries (no silent misassignment). The function returns a `Peptidoform` directly so callers don't double-parse for downstream fragment annotation; `format_proforma(peptidoform)` populates `PEPTIDE_TABLE.modified_sequence`. The inverse `format_encyclopedia_modseq(peptidoform)` is the writer-side helper — lossy on the chemical-fidelity axis because EncyclopeDIA cannot represent terminal vs side-chain placement (terminal mods collapse onto residue 0). The translation lives in the encyclopedia format adapter at `massspec.io.encyclopedia`, not in `core.chem` and not in `thirdparty/` — it's pure file-format translation, not a tool-jar binding.

## Conventions (massspec layer)

- **`Peptidoform.charge` is a serialization artifact, not a chemical property.** The ProForma 2.0 spec encodes precursor charge as a `/N` suffix on the modseq string (and adducts as `/N[+Na+,-H+]`), so a peptidoform parsed from a library record or search-engine output carries the published charge state alongside the residue sequence and modifications. The molecule itself isn't `+2` in solution — that charge state is acquired in the ESI source. `precursor_mz(peptidoform, *, charge=None, ...)` falls back to `peptidoform.charge` when the kwarg is omitted but lets explicit `charge=N` override (e.g. for charge-state grids over a charge-naive peptidoform). If we ever need to separate the format-shaped charge from the molecule-shaped charge, that's a `PrecursorRecord`-style wrapper on top, not a refactor of `Peptidoform`.
- **Spectral libraries split theoretical from empirical.** `massspec.library` carries the protein → peptide → precursor → fragment DAG plus the only true M:N edge (protein ↔ peptide) and entity-level *predicted* properties (RT / CCS / fragment intensity) — sample-agnostic, sequence-derivable. `massspec.quant` carries per-acquisition observations (Protein/Peptide/PrecursorQuant) and *calibrated* per-edge transmission efficiencies (`TransmissionProteinPeptide`, `TransmissionPeptidePrecursor`); `efficiency` is strictly bounded `(0, 1]` on populated rows with `-1.0` as the uncalibrated sentinel. `massspec.search` holds peptide-level + protein-level detection scores (q-values, posterior error probabilities, percolator scores) and is the third sibling — keyed by `(entity_id, acquisition_id, engine)`. Run provenance lives once in `massspec.acquisitions` (`ACQUISITION_TABLE` + `Acquisitions` container + `validate_acquisitions` cross-check); Library, Quant, and Search all reference `acquisition_id`. A higher-level composition layer that unifies Library + Search + Quant per Constellation's "detection-IS-quantification" philosophy is intentionally deferred until a concrete pipeline drives its API. Library/Quant/Search each ship Reader/Writer Protocols + registries with `ParquetDirReader/Writer` as the native lossless form. EncyclopeDIA `.dlib` / `.elib` (the same SQLite container — DLIB and ELIB differ only in which tables are populated) is implemented in `massspec.io.encyclopedia` and self-registers with all three Reader/Writer registries; one shared core produces a `(Library, Quant?, Search?)` tuple via `read_encyclopedia(path)` and slot-specific Protocols (`load_library` / `load_quant` / `load_search`) project out the requested view.
